# Vault Indexing & Retrieval — Deep Notes

Loaded on demand when working under `rag/`. Cross-cutting invariants (file types, chunk-ID scheme, reindex-free knob list, deletion auditing) live in the root `CLAUDE.md`; this file holds the implementation detail behind them.

## Indexing pipeline

- Indexing streams end-to-end: `_load_vault_documents` is itself a generator (M4) that yields one `LlamaDocument` at a time — MD docs first (sorted), then sorted referenced image descriptions, then PDFs in scan order. `_chunk_raw_documents` consumes that generator one document at a time, and `_index_documents_streaming` inserts chunks in **bounded embed batches** (Track 5.6, below). Peak RAM is bounded by the largest single source document plus the chunker/indexer backlog plus one embed batch, not by the full vault.
- **Batched embedding (Track 5.6, 2026-07-04).** Chunks needing (re-)embedding buffer up to `vault_embed_batch_size` (config-only, 1–256, default 16; ≤1 = the legacy per-chunk `idx.insert` path byte-for-byte) and flush as ONE `insert_nodes` call — `embed_nodes` hands the whole batch to `get_text_embedding_batch`, and the indexing embed model's `embed_batch_size` is set to the same knob (post-construction, best-effort) so a flush is one provider HTTP round-trip; on lancedb a flush is also one transaction instead of one per chunk. The batch path replicates `BaseIndex.insert()` exactly — `run_transformations` (outside the mutation lock — pure CPU; the legacy path parsed under it) → `insert_nodes` + per-doc `set_document_hash` under the lock — verified against llama-index-core 0.14.22, so chunk doc_ids/hashes are untouched (**reindex-neutral**). Failure semantics preserved: a stage-1 (transformation) failure degrades straight to the per-doc path (nothing written — this is also the path test fakes without `_transformations` take); a stage-2 (embed/store) failure runs an idempotency cleanup (`delete_ref_doc` per buffered doc, errors ignored) then retries chunk-by-chunk on the ORIGINAL per-doc path, so the consecutive-failure breaker, interruptible backoff, and `reinsert_failed` gap-tracking still count per chunk. `batch_target` is capped at `_PERSIST_EVERY` so a batch never straddles the checkpoint cadence (a defensive cap only — it never binds for shipping config, since the knob is ≤256 and `_PERSIST_EVERY` is 500; it exists for the test that patches `_PERSIST_EVERY` below the knob). A cancel — **or the consecutive-failure breaker tripping mid per-doc-fallback** — with deleted-old chunks still queued reports every one of them via `reinsert_failed` (the buffer opens a delete→insert window the adjacent legacy ops didn't have; the fallback's early-return path records the *unprocessed* remainder of the aborted batch too, so the gap count is complete — `TestVaultIndexingRegressions::test_batch_fallback_records_all_gaps_when_breaker_trips_midbatch`). **Concurrency trade-off:** the whole-batch embed round-trip in stage-2 runs while `_index_mutation_lock` is held (like K sequential inserts, but as one hold), so a larger `vault_embed_batch_size` lengthens the worst-case stall a *concurrent* single-shot chat's retrieval sees waiting on that lock — bounded by the 120 s `_acquire_retrieval_lock` (a clean error, never a hang), so it is a latency, not a correctness, knob. `TestEmbedBatchParity` also pins the lancedb batch path (heterogeneous MD/PDF/image metadata through one `insert_nodes`, projection intact) and that a flush larger than the embed model's default sub-batch (10) is one call only *because of* the `embed_batch_size` alignment (`[15]` aligned vs `[10,5]` not). Residual caveat: a provider may return *slightly* different numerics for batched vs single embeds — hermetic parity (identical docstore/hashes/vectors, `[3,2]` vs `[1]×5` calls, zero-call re-run, breaker bound) can't see that; validate with `tests/eval/` (`RUN_LIVE_EVAL=1`) before trusting retrieval quality on a new backend, or set the knob to 1.
- **Pruned scan walk (Track 5.4, 2026-07-04).** The scan pass uses `_iter_vault_scan_files` (an iterative `os.scandir` walk) instead of `sorted(vault_root.rglob("*"))` + a per-file `resolve()`: reserved-name dirs (`OBSIDIAN_EXCLUDED_DIR_NAMES`) and the user's `vault_exclude_dirs` are pruned at DESCENT time (their contents are never even listed — the old walk materialised and sorted the entire tree including `.git`/`.trash`/excluded folders, worst on iCloud), and no per-file symlink-resolving syscall chain runs. Equivalence: `vault_root` arrives resolved, so apparent rel == resolved rel for non-symlink entries; symlinked FILES keep the old resolve-gate (yielded only when the target stays under the root — a vault symlink cannot pull outside content into the index); symlinked DIRS are not descended (os.walk / Python 3.13 rglob semantics — under the old 3.12 walk their content either resolved outside the root and was skipped, or was indexed twice under both rels, a footgun not a feature). Per-type buckets are sorted post-walk, reproducing the old globally-sorted MD/PDF/name_index order byte-identically (subsequence of a sorted list). Pinned by `test_vault_regressions.py::TestVaultScanWalk` (descent-pruning spy + order + symlink gates).
- **MD secondary cap pass** (`_chunk_raw_documents`, MD branch only). `MarkdownNodeParser` splits `.md` at heading boundaries with no size ceiling, so one long single-heading section can exceed the embedding model's token limit (nomic-embed-text / EmbeddingGemma both ~2048) and be **silently truncated** — its tail never embeds. After the heading split, each section is run through a second `SentenceSplitter(chunk_size=MD_MAX_CHUNK_TOKENS=1024, chunk_overlap=64)` and **oversize is detected by output cardinality**: `>1` node ⇒ the section was over cap ⇒ use the sub-chunks; `<=1` ⇒ keep the **original** node object (not the splitter's output — the splitter may normalise whitespace; keeping the original guarantees byte-identity). A cheap, provably-safe fast path skips tokenizing tiny sections: a cl100k token is always ≥ 1 UTF-8 byte, so `len(text.encode()) <= cap` ⇒ token count ≤ cap (a char-count floor would be wrong for multibyte scripts). The secondary split is wrapped in `try/except` that degrades to the un-split section — `_chunk_raw_documents` is a generator, so an unhandled exception there would abort the whole run, bypassing `_index_documents_streaming`'s per-chunk insert guard. **Churn:** under-cap sections keep their `i`+text so doc_ids are byte-identical (zero re-embed for the ~97% of notes with no oversized section); a note *with* an oversized section re-hashes that section and everything after it (the `i` shift) — the stale-doc_id sweep self-heals it, **no `OBSIDIAN_INDEX_VERSION` bump** (it converges incrementally). Attachment extraction runs over the **final** (post-split) node list, so each sub-chunk records only the links in its own text, and the (potentially large) `attachments` list is attached **after** the split so it never inflates the split decision (the `_tags.md` metadata-budget case). PDF chunking is untouched (already 512-token capped). `MD_MAX_CHUNK_TOKENS` is a **pinned constant** (not a knob): it affects chunk IDs, so editing it is a reindex — same contract as `chunk_size=512`. Pinned by `test_chunker_params_pinned`; behaviour covered by `TestMdSecondaryCap`.
- The PDF signature-cache write lives in a `try/finally` inside `_load_vault_documents` so it runs whether iteration completes, the consumer breaks early, or the generator is closed by a test. The empty-vault diagnostic moved from "before chunking" to "after streaming" — if the streamer reports `added=skipped=failed=0` and `_stop_event` is not set, the run surfaces `ERROR: No indexable files found.` instead of writing a "done" meta with zero documents.
- Per-insert tolerance (now the per-doc FALLBACK path of the batched flush — same counters, same bounds): a single failed `idx.insert(doc)` increments a counter and continues. The run aborts after `_MAX_CONSECUTIVE_FAILURES` (20) consecutive failures with the partial index preserved. The breaker is a **wall-clock** window, not an instant count: each consecutive failure waits `min(_FAILURE_BACKOFF_BASE_S * 2**(streak-1), _FAILURE_BACKOFF_CAP_S)` (1.0 / 5.0 s → ~87 s of recovery window across the 19 sleeps before the 20th-failure abort) before the next attempt, via the interruptible `self._stop_event.wait(delay)` (a Cancel/Pause mid-sleep returns immediately and aborts, mirroring the top-of-loop stop check). This stops a backend that *instant-rejects* (e.g. LM Studio returning HTTP 400 in ~10 ms while it JIT-reloads the embed model under memory pressure) from burning all 20 "retries" in ~0.2 s and killing a multi-hour run before the model finishes loading; a streak-resetting success cancels the backoff. The sleep holds no lock (the insert `with self._index_mutation_lock` block has exited) and runs off the chat path (the rw write lock is released for the insert loop). `_FAILURE_BACKOFF_BASE_S`/`_FAILURE_BACKOFF_CAP_S` are class attrs patched to 0 in tests for instant runs (same pattern as `_PERSIST_MIN_INTERVAL_S`). Chunks dropped during a blip were never inserted, so the next incremental run re-yields and embeds them (self-healing — the breaker's job is to let the *run* survive, not to recover the individual chunk in place).
- **Checkpoint-promotion marker (2026-07-04, item 2.8a).** Promotion into `obsidian_storage/` replaces the store files one `os.replace` at a time — per-file atomic, NOT transactional across files. `_persist_index_checkpoint` now brackets the replace loop with `promotion_state.json` (fsync'd atomic writes: `"promoting"` before the first replace, `"complete"` after the last), and the two LOAD-time validations (`_ensure_index_loaded`, `prewarm`) pass `check_promotion_marker=True` — a marker still saying `promoting` refuses the load with a clear error into the existing integrity-error recovery instead of silently serving a mixed-generation store whose stranded chunks the hash-skip would never re-insert. A missing marker = pre-marker legacy checkpoint, accepted. Guards the process-crash/SIGKILL window; power-loss rename reordering is out of scope (same stance as the atomic writers' no-parent-fsync note). **The WRITER path honors it too (2026-07-05 follow-up):** `index_vault`'s incremental branch gates `is_incremental` on `_incremental_store_is_intact(prev_meta)` (the same `_validate_persisted_index_files(check_promotion_marker=True, full=False, require_vector_data=False)` check the readers run) — without this the documented recovery ("Re-run indexing to rebuild a consistent checkpoint") would *load the torn store incrementally* (the hash-skip never repairs stranded chunks) and then stamp a fresh `"complete"` marker over it, permanently hiding the corruption from every future load-time check; now a torn store degrades to a fresh rebuild, so the recovery actually rebuilds. Pinned by `TestCheckpointPromotionMarker` + `TestReindexHonorsPromotionMarker`.
- **Stale-sweep read-failure protection (item 2.8b).** The loader records per-run read/extract failures (`_scan_failed_sources`: MD read, PDF extract, vision description); the stale-doc sweep withholds any `previous - current` doc_id whose `{rel}::` prefix matches a failed source — a transiently unreadable file (iCloud dataless placeholder, I/O blip) costs one run of staleness instead of a silent de-index. Genuinely deleted files sweep normally; withheld ids resync next run. Pinned by `TestStaleSweepSparesScanFailures`.
- **Final-persist cache invalidation runs OUTSIDE the rw write lock (item 2.8c)**, matching the mid-run checkpoint path (the two had drifted): inside the lock, `_invalidate_retrieval_caches` contended the BM25 build lock while every reader queued on the write lock — all chats froze for the duration of a concurrent BM25 rebuild. Same eventual-consistency contract as mid-run (staleness until the invalidation lands, never corruption). Pinned by `TestFinalPersistCacheInvalidationLockOrder`.
- Mid-run checkpoint: once `_PERSIST_EVERY` (500) successful inserts have accumulated **and** `_PERSIST_MIN_INTERVAL_S` (600 s) has elapsed since the previous checkpoint attempt, the index, meta, and manifest are written. Both gates must hold — each checkpoint re-serialises and re-validates the entire store with the GIL held while chat is blocked on the rw write lock, so a count-only cadence made fast embedding runs spend a growing fraction of their time inside O(N) dumps. Crash re-work is bounded by max(500 chunks, ~10 min); patch `_PERSIST_MIN_INTERVAL_S` to 0 in tests for count-only behaviour. LlamaIndex storage is persisted through `_persist_index_checkpoint`: write to a sibling temp directory, validate the persisted checkpoint files, then promote them into `obsidian_storage/`. **The checkpoint validates with `full=False`** (W3, 2026-07-01): it runs the cheap structural tail-check (`_json_file_has_complete_tail`) + the vector-data presence check but **skips the full `json.load` sweep** — that sweep re-parsed the ~3 GB `default__vector_store.json` (simple backend) into a throwaway graph on every checkpoint under the write lock. Safe because json.dump writes a complete document or the tail-check catches a truncated one, the read-back is page-cache-served (so the full parse never validated on-disk bytes anyway), and the authoritative parse still runs at **load** (`_ensure_index_loaded`/`prewarm`, then `_build_index_for_backend` → JSONDecodeError → integrity error → `paused_scan`). `_write_json_atomic` (meta/manifest/signatures — low frequency) fsyncs before its rename; the per-document regenerable description cache (`_atomic_write_text`) deliberately does not.
- The streaming indexer accumulates the manifest as `{raw_source: chunk_count}` (`_manifest_source` keys it by `metadata.source`/`file_path`, with a `""` bucket for sourceless docs) instead of a list of full `LlamaDocument` objects — the old list pinned a duplicate copy of every chunk's text (and its pydantic overhead) for the whole run, dominating peak RAM on incremental re-runs where most chunks are skipped. `_write_index_manifest(counts, vault_path)` resolves each unique raw source once and skips the `""` bucket; `index_vault` derives `has_vector_data` / `inserted_this_run` from the dict (`bool` / `sum`), so the manifest JSON and meta stay byte-identical to the list era.
- On startup `get_status()` reads `obsidian_meta.json` from disk and recovers `"done"`, `"paused_partial"`, or `"paused_scan"` for an index state that already exists on disk. A paused-during-scan run writes meta without ever persisting `docstore.json`, so meta alone is sufficient to recover that case. If meta claims vector data but the checkpoint files are missing or have an incomplete JSON tail, the state falls back to `"paused_scan"` and `integrity_error` is included in the status payload. Chat lazy-loads run the same cheap preflight and wrap any load-time JSON corruption in a clearer checkpoint error.
- Large PDFs (> `EXTRACT_MAX_PAGES_PER_CALL` = 1000 pages) become **one `LlamaDocument` per 1000-page range** via `_load_pdf_range_documents`: each range is extracted, cached, and yielded independently, so peak RAM is bounded by ~1000 pages of text regardless of file size, a cancel keeps every completed range's cache, and a resumed run continues from the last cached range. Range documents carry `page_start` / `page_end` metadata; `_chunk_raw_documents` salts the chunk hash with `page_start` (`sha1(f"{page_start}:{i}\n{text}")`) so two ranges of the same file can never collide on a doc_id — documents *without* `page_start` keep the original unsalted input, which is the no-reindex guarantee for everything else. PDFs over `PDF_MAX_PAGES` (= 20 000, `core/constants.py`, shared with the single-paper upload worker) are skipped with a warning — a time bound, not a memory bound.
- **Grandfathering:** large PDFs whose text was cached as one concatenated file by the pre-per-range loader are still served from that whole-file cache as a single document, so their chunk IDs (and stored embeddings) stay valid. Deleting the cache file opts the PDF into per-range documents on the next run. Per-range caches live beside the whole-file cache as `{sha256}-pSSSSS-EEEEE.txt` (zero-padded so lexicographic order = page order); `read_note`'s `_read_pdf_text` stitches them back together, reporting `truncated=True` when the cached ranges don't cover the whole document.
- `_pdf_file_signature` consults `pdf_signatures.json` and reuses a previous sha256 when size+mtime are unchanged, avoiding a full re-hash of large textbook PDFs on every resume. The **read paths** reuse it too: `_read_pdf_text` (read_note / agent `vault_read_note`) and `refactor/pdfref.py::_has_cached_text` pass `_persisted_pdf_signatures()` — a stat-cached, read-only in-memory view of the persisted map (keyed by the signature files' `(size, mtime_ns)`, so an indexing run's rewrite refreshes it on the next call) — instead of re-hashing the whole PDF per call just to locate its extracted-text cache. Only the indexer writes the persisted map; a read-path cache miss still falls back to a full hash without writing back.
- When `OBSIDIAN_INDEX_VERSION` changes, the prior vector index dir is renamed to `obsidian_storage.bak.{prev_version}.{timestamp}/` only if it contains real vector/docstore data.
- PDF extraction and image descriptions are cached under `obsidian_cache/`, split by vault key and content hash. Legacy cache files under `obsidian_storage/` are still read and are copied out before a vector-index archive.
- `obsidian_meta.json` and `indexed_materials.json` are written atomically via the `_write_json_atomic` helper in `rag/vault.py` (same pattern as the PDF cache). Index metadata stores the embedding model; `indexed_materials.json` stores the vault-relative files and chunk counts shown in the Indexed Materials panel. The UI shows a warning when the current embedding model does not match the index metadata. Beyond that advisory banner, **retrieval enforces it**: `stream_chat`/`retrieve` resolve the index's recorded embed model via `_effective_embed_name` and query with **it** (warning once via `stage_cb`) whenever the configured `embed` differs — so a model switch without a reindex cannot fuse two vector spaces and silently wreck results. The indexer separately forces a fresh rebuild when the embed model changes, or when an existing `has_vector_data` index recorded **no** embed model (can't prove compatibility).
- Images: referenced images that exceed the 20 MB cap, fail to read, or come back from the vision model with an empty description are counted under `skipped_image_count` and surfaced in the status payload. Unreferenced images are ignored. GLM-OCR retries scanned PDF pages with a smaller aligned image when the backend reports a context-window overflow.
- **Scanned-PDF OCR coverage.** `_perform_ocr_fallback`'s default cap is `_OCR_MAX_PAGES` (100) **per extraction call**, but the vault loader extracts in 1000-page ranges — so the indexer passes `ocr_max_pages=EXTRACT_MAX_PAGES_PER_CALL` to OCR the whole range (the un-overridden default would silently index only the first 100 pages of a scanned ≥100-page range and cache that partial as complete). It also passes `page_done_cb` (a per-page `_op_lock.heartbeat`) so a long scan can't expire the 3600 s op-lock TTL. When OCR still can't cover the requested range, `ArticleSections.truncated` is set and the loader **warns and skips caching** that range so it retries next run instead of permanently serving a partial. The interactive single-paper upload path keeps the 100-page default (bounded by its own `PDF_EXTRACT_TIMEOUT_S`).
- Image link resolution mirrors Obsidian. `_resolve_md_attachment` first tries the target relative to the linking note's folder (an existing file there wins); a **bare** filename that misses there falls back to a vault-wide basename lookup against an index `_load_vault_documents` builds during its single `rglob` walk (`basename.lower() -> [vault-relative path]`, non-excluded image files only), so `![[image.png]]` resolves even when the image lives in a central attachments folder. Ambiguous basenames are broken by `_pick_closest_attachment` (longest shared dir prefix with the note, then fewest segments, then lexicographic). The `is_file` probe and the basename fallback are gated on `name_index is not None`, so they run only on the image-load path; the `attachments`-metadata call in `_chunk_raw_documents` passes no index and returns the parent-relative path verbatim with zero stats (it is read by tests only, and changing it would alter the `MetadataMode.ALL` document hash and force re-embedding).
- Two link shapes the basename fallback deliberately does **not** cover (neither is a regression — the pre-existing resolver handled them no better): (1) a **partial-path** wikilink such as `![[subdir/img.png]]` pointing at a central folder — anything with a `/` skips the fallback and is resolved parent-relative only, so it must already sit at that path below the note; (2) **basename case-collisions on a case-sensitive filesystem** — the index key is `basename.lower()`, so `Fig.png` and `fig.png` merge into one bucket and `_pick_closest_attachment`'s tie-break may pick the other case. Both are non-issues for the common case (macOS case-insensitive FS, bare-filename links), which is why the fallback keys on the lowercased basename in the first place.
- The vault-image vision prompt (`VisionManager.describe_image`, the **only** non-test caller is `_extract_image_description`) asks for a short description of what the image depicts **plus** a transcription of any visible text/labels/data — not the pure "extract all text" OCR prompt the scanned-PDF path (`GLMOCRManager.extract_page_text`) still uses. Pure-OCR returned nothing for text-light figures, so they embedded empty and were dropped. The image-description cache is keyed by image bytes (not prompt text), so changing the prompt does **not** invalidate already-cached descriptions — clear `obsidian_cache/.../image_cache/` to regenerate them.

## Vector store backends (Batch 4)

- Two backends: the legacy JSON `SimpleVectorStore` (default) and binary **LanceDB** (`rag/lancedb_store.py`). The active backend for an existing index is read from `obsidian_meta.json` (`vector_backend`; absent ⇒ `simple`) and is **authoritative** — never overridden by the `vault_vector_backend` config knob, which only chooses the backend for a *fresh* build (and degrades to `simple` if lancedb is not importable). All four index build/load sites — fresh build, incremental load (`index_vault`), lazy chat load (`_ensure_index_loaded`), and `prewarm` — go through `_build_storage_context` / `_build_index_for_backend`.
- **`store_nodes_override=True` is mandatory on the lancedb path.** LanceDB stores node text (`stores_text=True`), so without the override LlamaIndex would stop writing nodes into the docstore on incremental runs, breaking BM25 (which builds from the docstore) and the document-hash skip-check. With it, `docstore.json` still holds every node — so the docstore/index-store JSON are still parsed at load and rewritten at each checkpoint (only the ~3 GB *vector* JSON disappears; Batch 1b's time-gate still matters).
- **`NormalizingLanceDBVectorStore`** wraps the integration so `rag/engine.py` and the embed model need no backend branches: `add()` unit-normalizes each node's embedding and JSON-stringifies list/dict metadata on a per-node copy (the docstore node keeps the original list); `query()` unit-normalizes the query vector. LanceDB's default search metric is L2, and for unit vectors L2 ordering is monotonic with cosine, so retrieval ranking is bit-for-bit identical to `SimpleVectorStore` cosine. (The returned dense *score* is the integration's `exp(-distance)`, a different scale than raw cosine — only relevant to the dense-only-no-reranker `similarity_cutoff`; RRF uses ranks, so fusion is unaffected.)
- **Metadata projection (schema-drift guard).** The integration stores node metadata as one LanceDB struct column whose field set is **frozen at table creation** (the vault table is created MD-first, so its user keys are `{file_path, source, extension, header_path, attachments}`); any later insert whose metadata adds a field the struct lacks fails with `field 'X' does not exist in table schema`. That silently broke large-PDF range chunks (`page_start`/`page_end`) and vault-image chunks (`is_image`) — large textbooks never indexed on lancedb, and each run aborted on the consecutive-failure breaker at the first one. `add()` therefore **projects** each per-node metadata copy onto the keys the persisted table actually has (`_allowed_metadata_keys()` reads the live struct schema; `_LANCE_FLAT_METADATA_KEYS` is the fresh-table fallback), dropping unknown keys so an extra-field insert is structurally impossible. Dropping is lossless for the read path: those keys are write-only at query time (`page_start` is consumed only at chunk-hash time in `_chunk_raw_documents`; `page_end`/`is_image` are unread), and the full node — every key — still lives in the docstore (`store_nodes_override`) and the row's `_node_content`. The projection is **vector-row only**, so `doc_id`/`doc.hash`/the docstore are untouched → already-indexed MD and PDF chunks still skip by hash with **no re-embedding**; only the previously-failing large PDFs and the images embed (first-time) on the next incremental run. Adding a new *filterable* column here is a deliberate, migration-requiring change — extending the projection alone would reintroduce the drift error on existing tables. Migration (`scripts/migrate_vector_store.py`) bulk-loads on its own path and does **not** apply this projection (out of scope; the live indexer is the only writer that mixes key sets).
- **Fresh-build column set is first-chunk-dependent (known, harmless).** A fresh build starts from `VectorStoreIndex.from_documents([])` and then streams **single-node** inserts, so the struct schema is frozen by the *first* chunk's keys. If the first MD chunk has no `attachments`, the table is created without an `attachments` column and the projection then *drops* `attachments` from later rows rather than storing it (vs. the pre-fix crash). This loses nothing the read path uses (`attachments` is read by tests only and stays in the docstore + `_node_content`); the only keys retrieval reads — `source`/`file_path` — are universal, so they are always present and never dropped. Net: whether `attachments` exists as a queryable column on a fresh lancedb index is non-deterministic, but nothing queries it.
- **Checkpoint:** on lancedb the vector data is written transactionally into `obsidian_storage/lancedb/` during `idx.insert` (durable immediately), so `_persist_index_checkpoint` promotes only the docstore/index-store JSON and `_validate_persisted_index_files` skips the JSON-vector-store check (the temp dir has no `default__vector_store.json`); `require_vector_data=False` there because the table lives in the live dir, not the temp one. `_index_dir_has_vector_data` checks a populated `lancedb` table (`lancedb_table_count > 0`) instead of `_simple_vector_store_has_any_embedding`. The `lancedb/` dir lives inside the index dir, so `/api/reset` and the version-bump archive cover it automatically.
- **Compaction (version-bloat guard).** Each streaming `idx.insert` is its own LanceDB transaction → a new single-row data fragment **and** a new version manifest that re-lists every fragment, so `_versions/` grows ~O(n²) (66k inserts measured at 209 GB over <1 GB of vectors). `lancedb_store.compact_lancedb_vector_store` (`Table.optimize(cleanup_older_than=timedelta(0))`, best-effort/never-raises) merges the fragments and prunes superseded versions, bounding `_versions/` to O(n). It runs (1) inside `_persist_index_checkpoint` (lancedb branch — already under `_index_mutation_lock` + rw write lock) and (2) every `_LANCEDB_COMPACT_EVERY` (2000) inserts in `_index_documents_streaming`, under `_index_mutation_lock` only (it changes neither query results nor the docstore, so it's much lighter than a checkpoint). The insert-count cadence matters because the JSON-checkpoint gate (`max(500, 600 s)`) is too coarse on a fast embedder — the first checkpoint can be ~30k inserts in, accumulating tens of GB of interim manifests before it fires. `scripts/compact_lancedb.py` is a one-off offline reclaim for a table already bloated by a crash before the first compaction.
- **Crash-drift recovery:** the binary store is durable-on-insert but the docstore persists at `max(500 chunks, 10 min)`, so a hard crash can leave the table *ahead* of the docstore (orphan rows for nodes the checkpoint never recorded). At incremental load, two O(1) counts (`lancedb_table_count` vs `len(docstore.docs)`) detect this; only when the table is ahead does the run set `lancedb_upsert=True`, which extends the changed-chunk delete-before-insert (`idx.delete_ref_doc`) to the *new*-chunk path so a re-processed chunk replaces its orphan row instead of duplicating. Normal runs and fresh builds do nothing. When the table is ahead, the run also sweeps **orphan rows** (ids present in the table but absent from the docstore — deleted-note leftovers) via `lancedb_list_doc_ids` (id-column projection through `to_lance()`, never a full `to_arrow()` materialisation) + `lancedb_delete_ids` (single-quoted SQL literals with `''` doubling — French filenames make apostrophes the common case — batch delete with per-id fallback, `(ok, detail)` return surfaced as a WARNING on failure). Residual gap: offsetting add/delete drift balances the counts so the branch never fires (root CLAUDE.md Known Issues).
- **Migration:** `scripts/migrate_vector_store.py` joins `embedding_dict` (streamed with `ijson`) to `docstore.json` nodes by id and bulk-loads LanceDB with **no embedding model constructed** — so node IDs/hashes are untouched and a subsequent incremental run reports `added=0` (the no-re-embed gate). It archives the JSON to `.bak`, writes `vector_backend=lancedb`, and is idempotent.

## Lock discipline

- **Op-lock epoch is a per-operation token, never shared state** (2026-07-04, item 1.5). `try_acquire_lock(ttl)` returns the acquisition epoch (`int`, ≥1, truthy; `None` when refused) — **captured atomically** inside `RagOperationLock.try_acquire_epoch`'s single `_meta_lock` critical section (2026-07-05 audit m2: the earlier two-step `try_acquire()` then a separate `.epoch` read left a window where a `force_release` + re-acquire between the two lines could hand back a *newer* acquisition's token) — and `release_lock(epoch)` / `heartbeat(epoch)` require it back; `index_vault(..., op_epoch=)` threads it into the loader/streaming-insert heartbeat sites. The epoch must live on the operation's own stack — the previous manager-level `_lock_epoch` attribute meant any new acquisition overwrote the token a still-running previous holder later released with (cancel an index run → start a refactor Apply → the indexer's `finally` released the *refactor's* lock mid-batch). `release_lock` refuses a falsy epoch (an unconditional release stays exclusive to `force_release`), and `heartbeat(0)`/`op_epoch=0` (a direct test call holding no lock) is inert. Invariant, pinned by `test_concurrency.py::TestOpLockEpochToken`: a holder can only release/extend the acquisition whose epoch it captured at acquire time. The refactor write routes bind the token into their zero-arg heartbeat callables (`heartbeat=lambda: obsidian_manager.heartbeat(op_epoch)`), so `refactor/`'s callback contract is unchanged.
- `stream_chat` uses double-checked locking: the lazy index load happens under a write lock so concurrent chat requests cannot both write `self._index` inside a read lock.
- **`_build_engine` is the single retrieval-engine build (item 4.8, repaired 2026-07-04).** `stream_chat` and `retrieve` both call `ObsidianVaultManager._build_engine(...)` — effective-embed resolution, lazy index load, BM25/reranker/wikilink/thesaurus builds, and the read-locked `SimpleQueryEngine` construction live there exactly once, and the callers only wrap `engine.query()`/`engine.retrieve()` in `_acquire_retrieval_lock`. The first 4.8 cut (d06cd73) shipped a self-recursive stub that deleted `stream_chat` outright — vault chat died on the first message; the repair restored the pre-d06cd73 build body verbatim inside the extraction. `retrieve` now accepts the thesaurus/primer knobs too (defaults off; the agent's `vault_search` never passes them, so the documented no-thesaurus-on-active-search contract is unchanged).
- The indexer publishes `self._index` after the setup phase and releases the write lock for the insert loop, so chat queries against the in-progress index do not block on multi-hour runs. The write lock is reacquired only for mid-run checkpoints and the final persist.
- `docstore_doc_count()` is the health probe's read-only accessor: a NON-blocking `_index_mutation_lock` acquire snapshotting `len(_index.docstore.docs)` (`None` = not loaded or briefly busy) — `/api/health` must never stall behind (or stall) an insert/checkpoint, and reading `_index` internals without the lock raced the streaming indexer.
- A dedicated `_index_mutation_lock` (`threading.Lock`) serialises three operations that touch the LlamaIndex internal data structures: `idx.insert` / `idx.delete_ref_doc` in the streaming indexer, `_persist_index_checkpoint` during mid-run and final checkpoints, and the retrieval phase of `engine.query()` in `stream_chat`. The LLM streaming runs outside the lock because `query_engine.query(message)` with `streaming=True` returns after retrieval.
- **Chat-path lock + embed bounds (2026-07-04, item 2.1).** The retrieval phase embeds the query over HTTP *while holding* `_index_mutation_lock`, so it is double-bounded: (1) the query-path embed model is constructed with a per-call HTTP timeout (`Provider.get_embedding(..., request_timeout_s=QUERY_EMBED_TIMEOUT_S)`, 30 s — Ollama via `client_kwargs={"timeout": …}`, LM Studio via `timeout=…, max_retries=0`) so a wedged backend call cannot hold the lock forever; (2) `stream_chat`/`retrieve` acquire the lock through `_acquire_retrieval_lock()` — a 120 s timed acquire (`_RETRIEVAL_LOCK_TIMEOUT_S`) raising an actionable error instead of blocking indefinitely. 120 s sits **above** the longest legitimate holder (a mid-run checkpoint re-serialising a multi-GB simple-backend store) and **below** the SSE consumer stall floor, so the clean error beats the generic "Generation timed out". Indexing embeds are constructed *without* `request_timeout_s` and stay deliberately unbounded — never default that parameter. Pinned by `TestQueryEmbedBound`. `/api/obsidian/chat` emits `info` SSE events for partial-index warnings and major chat stages (loading index, building BM25, loading reranker, retrieval/model start).

## Prewarm

`ObsidianVaultManager.prewarm()` is invoked from a daemon thread in `launch.py` once Flask is up. It walks the same load path that `stream_chat` would otherwise hit lazily on the first query — index from disk, then BM25 (if `vault_hybrid_enabled`; a valid BM25 sidecar makes the `building_bm25` stage a fast mmap open, and a fresh build persists the sidecar here, off the chat path), then the cross-encoder reranker (if `vault_reranker_enabled` and `vault_reranker_model` is set) — so cold-disk reads of the docstore (~620 MB) and vector store (~3 GB) do not steal the chat-token timeout. Progress is exposed via `prewarm_status` / `prewarm_message` in `/api/obsidian/status` and rendered in a banner above the chat input (`#prewarm-banner` in `templates/index.html`); the Send button is disabled while a prewarm is in flight. States: `idle` (not started), `loading_index`, `building_bm25`, `loading_reranker`, `ready`, `skipped` (nothing on disk), `error`. `prewarm()` is idempotent on a `_prewarm_started` flag; `reset_prewarm()` clears it and is called from `cleanup()` so `/api/reset` allows a subsequent warm to run. `vault_prewarm_enabled: false` short-circuits `prewarm()` to a terminal `skipped` ("Prewarm disabled in settings.") before any disk access — the knob defers the multi-GB load to the first chat rather than removing it. The indexer also marks prewarm as `ready` immediately after publishing `self._index`, so users who reach a populated index via reindexing (not via launch prewarm) get a cleared banner without waiting for prewarm to complete.

## Retrieval mechanics

- Vault chat retrieval is hybrid by default: BM25 (lexical) and dense (cosine) retrievers run in parallel and are fused via reciprocal-rank fusion (`QueryFusionRetriever(mode="reciprocal_rerank")`). **Leg weights** (`vault_rrf_dense_weight` / `vault_rrf_bm25_weight`, config-only, 0.0–10.0, default 1.0): LlamaIndex ignores `retriever_weights` under `reciprocal_rerank` (only `_relative_score_fusion` reads them), so `_fusion_mode_and_weights` keeps RRF byte-identical at the 1.0 defaults and switches that query to `mode="relative_score"` — the one mode that applies weights — when any weight differs. The two modes score on different scales (RRF: rank sums; relative_score: min-max-normalised leg scores blended by weight), which also changes what the no-reranker `similarity_cutoff` sees — weight changes should be validated against `tests/eval/`. Garbage/hand-edited weight values degrade to neutral 1.0 (`_fusion_weight`). `num_queries` defaults to `1` — a thin RRF wrapper with no LLM-driven query rewriting; when `query_expansion` is enabled it rises to `num_queries` (1-5) and the fusion retriever uses the per-query LLM to rewrite the question into variants and RRF-fuse their results (effective on **local** providers only — the online chat path has no llama-index LLM object, so it stays single-query there, and a fusion retriever is also constructed for the single-dense leg when expansion is on but hybrid is off). The per-query LLM is still passed explicitly via `llm=` — without it, `QueryFusionRetriever` falls back to `Settings.llm` whose lazy default is OpenAI and surfaces as a spurious "No API key found for OpenAI" error during chat, even when `num_queries=1` means the LLM is never actually invoked.
- Optional **MMR** diversity on the dense leg (gated by `mmr_enabled` with `mmr_lambda` as the threshold) de-duplicates near-identical chunks; no reindex. On `SimpleVectorStore` it uses LlamaIndex's native `vector_store_query_mode="mmr"` over the stored vectors. LanceDB silently ignores that query mode, so on the lancedb backend `rag/engine.py` detects the store (`is_lancedb_store`), has the dense retriever over-fetch a larger candidate pool, and applies MMR client-side (`_ClientSideMMRRetriever`, via `get_top_k_mmr_embeddings`) over the nodes' stored (normalized) embeddings — an approximation of the native dense-leg semantics, not bit-identical, but it keeps the knob working on the binary backend. A cross-encoder reranker (default `cross-encoder/ms-marco-MiniLM-L-6-v2`, ~67 MB) then narrows the candidate pool down to `top_k` before chunks reach the LLM. All stages are togglable via the `vault_*` config keys or the matching per-request body fields, and are query-time-only — toggling takes effect on the next Send without a reindex.
- Candidate pool size for the rerank stage is `min(max(top_k * multiplier, floor), ceiling)` (see `SimpleQueryEngine._rerank_pool_size`), where `multiplier` / `floor` / `ceiling` default to `4` / `20` / `50` and are overridable via the `vault_rerank_pool_multiplier` / `vault_rerank_pool_floor` / `vault_rerank_pool_ceiling` config keys; the ceiling is additionally a live per-request override (`rerank_pool_ceiling`, 10-200, body wins over config). When the reranker is off, retrieval breadth equals `top_k` exactly (the pre-2026-05-14 behaviour). The `_effective_top_k` autoscaler always applies to the final post-rerank count, never to the candidate pool — a small-context model is not overwhelmed by reranker breadth because only the final chunks reach the LLM.
- `SimilarityPostprocessor(similarity_cutoff=...)` is attached *only when the reranker is absent*. A cross-encoder rerank score is on a different scale to dense cosine, so a 0.25 cosine cutoff would silently drop high-rerank-score chunks. With hybrid retrieval but no reranker, the cutoff is applied to RRF fused scores (different semantics — keep in mind if the UI slider is re-exposed in that mode).
- When `top_k` is set explicitly (came from a request body or config rather than the function default), `SimpleQueryEngine` skips the `_effective_top_k` autoscaling and trusts the user's choice as-is; the autoscaler still applies when callers leave `top_k_explicit=False`.
- **Wikilink graph expansion** (`vault_wikilink_expansion`, default off; query-time, no reindex). When enabled, `_build_retrieval_pipeline` wraps the finalized retriever (dense or fusion) in `_WikilinkExpansionRetriever` **before** the postprocessor stack — one insertion point that covers the local, online, and agent-search paths since all three pull the single shared `retriever`. The wrapper runs the inner retriever for seeds, then for each seed note appends docstore chunks from its neighbour notes (outbound links ∪ backlinks, from the `_WikilinkGraph`) at a decayed seed score (`vault_wikilink_score_decay`), deduped, bounded by `vault_wikilink_neighbor_cap` (distinct neighbour notes) and `vault_wikilink_node_cap` (total appended chunks). Sitting **before** the rerank stage makes it rerank-gated: the cross-encoder re-scores seeds + neighbours uniformly and trims to the final top-k, so an off-topic neighbour drops out and the answer size is unchanged. It is therefore **only attached when a reranker is active** (`self.reranker is not None` in `_build_retrieval_pipeline`) — the reranker's `top_n` trim is the only thing bounding the post-expansion count, so without it expansion would push up to `node_cap` chunks past `top_k` at the LLM and risk overflowing a small context window. With no reranker the engine skips the wrap and `stream_chat`/`retrieve` skip the graph build entirely (no-op, no wasted work). The decayed seed score (`vault_wikilink_score_decay`) only sets the neighbours' initial ordering into the reranker's input; the reranker's own scores are authoritative. The graph (`_get_wikilink_index`, `rag/vault.py`) is built lazily from a docstore snapshot taken under `_index_mutation_lock`, cached by docstore size, and dropped by `_invalidate_retrieval_caches` alongside BM25 — same discipline, including an on-disk sidecar (see *Retrieval caches*). Resolution is FS-free: bare `[[note]]` links resolve against the in-memory set of indexed `.md` sources (Obsidian shortest-path, `_closest_note_by_dir` tie-break), never the filesystem. Notes already present as seeds are not re-expanded. lancedb-safe (reads full nodes from the docstore, immune to the vector-row schema freeze).
- **Thesaurus query expansion + abbreviation primer** (`vault_thesaurus_expansion` / `vault_primer_enabled`, both default off; query-time, no reindex, no re-embed). The thesaurus is parsed from two hand-curated tables whose **vault-relative paths are config-driven**: `vault_thesaurus_abbrev_path` (default `_abreviations.md`) and `vault_thesaurus_tags_path` (default `_tags.md`); `""` disables a slot. `_get_thesaurus()` resolves each under the vault root via `_resolve_thesaurus_rel` (rejects traversal/absolute/escaping — defence-in-depth over the `/api/config` `_coerce_vault_rel_file` shape check), reads + parses them lazily, cached by the **resolved paths plus** each file's `(size, mtime_ns)` signature (an edit to a file *or* a config path change refreshes on the next chat) and cleared on vault switch / `cleanup()`; it returns `None` (disabling both features) when neither resolved file exists. (`_THESAURUS_DEFAULT_ABBREV` / `_THESAURUS_DEFAULT_TAGS` are just the fallback names.) The parser/consumers live in `rag/thesaurus.py` (app-independent, stdlib-only). **Query expansion**: when `thesaurus_expansion` is on and a thesaurus is present, `_build_retrieval_pipeline` wraps the finalized retriever in `_ThesaurusExpansionRetriever` (`rag/engine.py`); for each retrieval it calls `Thesaurus.expand_query(message, max_variants)` (config cap `vault_thesaurus_max_variants`, 1-8/3), substituting one matched concept/abbreviation per variant with a known synonym (longest phrase first, accent/case-insensitive whole-token matching, ≥2-char floor, collision-flagged `≠` and morphological `...q` rows excluded), retrieves each variant **separately** (no dense-embedding dilution; lexical BM25 can hit the synonym token) and RRF-fuses. Purely additive — no thesaurus / no variants / empty query falls back to the seeds. **Primer**: when `primer_enabled` is on, `SimpleQueryEngine._build_primer` calls `Thesaurus.build_primer(message, vault_primer_max_chars)` (500-8000/1500) to render a compact glossary block (query-relevant rows first so truncation drops generic core entries last) which `_apply_vault_primer` prepends to the answer-mode template (online path: routed via `LLMRequest.system_prompt`). Both share the single cached `Thesaurus` object (`stream_chat` builds it once when `thesaurus_expansion or primer_enabled`) and are best-effort: `expand_query`/`build_primer` failures are swallowed and degrade to seeds-only / no-primer rather than crashing the chat. Both apply to single-shot RAG and the agent's RAG fallback, **not** the agent's active `vault_search` (parity with `wikilink_expansion`/`mmr`/`query_expansion`). **Domain defaults (overridable):** `rag/thesaurus.py`'s built-ins are tuned for the maintainer's bilingual FR/EN psychiatry/clinical-research vault, but the corpus-specific pieces are config-driven — `vault_primer_header` overrides the `_PRIMER_HEADER` intro and `vault_primer_core_terms` (comma-separated, read in `_build_primer`) overrides the `_CORE_PRIMER_TERMS` priority list (both empty ⇒ built-ins), threaded into `build_primer(..., header=, core_terms=)`; the parser also accepts English headers (`Abbreviation | Meaning`, `Description | Tag`). The `≠` collision marker stays a convention. Defaults-off, so a vault without the curated files is unaffected. Covered by `test_thesaurus.py`.
- `system_prompt` is a user-supplied **prefix** layered over the selected Answer Mode template via `rag.engine._apply_custom_prefix`. The base template's safety preamble, untrusted-context guard, and `{context_str}` / `{query_str}` placeholders remain app-controlled, so a typo cannot disable retrieval grounding. On the online path the prompt is routed through `LLMRequest.system_prompt` so provider adapters use their native system field rather than baking it into the user turn. Empty string ⇒ use the mode template as-is.

## Retrieval caches

- `ObsidianVaultManager._get_bm25_retriever(top_k)` caches the BM25Retriever by docstore size. The cache is rebuilt automatically when the size changes and explicitly invalidated by `_invalidate_retrieval_caches()` after every successful `idx.storage_context.persist(...)` (both mid-run checkpoint and final persist) and inside `cleanup()`. Same-size delete+insert is tolerated as eventually-consistent; the indexer's explicit invalidation resyncs.
- **BM25 sidecar** (`obsidian_storage/bm25_index/`): a successful in-process build is persisted in bm25s' on-disk format (`_persist_bm25_sidecar`) and the next process-cold miss mmap-loads it (`_load_bm25_sidecar`, `BM25Retriever.from_persist_dir(..., mmap=True)`) — score arrays are memory-mapped and the corpus is served lazily from JSONL, so steady-state RAM no longer holds a second full copy of all vault text and launch/first-chat skips the minutes-long re-tokenise. Staleness checks: sidecar `sidecar_meta.json` must match the live docstore count **and** the index meta's `indexed_at` stamp (count alone could accept a same-size-different-content sidecar left by a crash between a checkpoint's persist and its invalidation); the loaded corpus length is re-verified and `similarity_top_k` re-applied (`from_persist_dir` restores the persisted value). Any failure deletes the sidecar and rebuilds from the docstore — deleting it by hand is always safe. Persisting is skipped while an indexing run is active (`_index_state` not `idle`/`done`) because `_invalidate_retrieval_caches()` — which also rmtrees the sidecar — fires at every mid-run checkpoint; the meta file is written last so a torn persist self-invalidates. The sidecar lives inside the index dir so `/api/reset` and the version-bump archive cover it; `_persist_index_checkpoint` promotes only `_LLAMAINDEX_PERSIST_FILES` and never touches it. `bm25s` is pinned to the verified minor line in `requirements.txt` because the sidecar is a serialized format.
- **Wikilink sidecar** (`obsidian_storage/wikilink_sidecar.json`): the BM25 sidecar's sibling for the note→note graph. A successful in-process build is persisted (`_persist_wikilink_sidecar`) and a process-cold miss in `_get_wikilink_index` loads it (`_load_wikilink_sidecar`) instead of copying the node list and regex-sweeping every `.md` chunk. One atomic JSON file (no dir + meta-last dance — `_write_json_atomic` makes a torn write impossible): `{doc_count, indexed_at, created_at, forward, note_to_node_ids}`; backlinks are deliberately **not** stored (`_WikilinkGraph.from_payload` re-inverts `forward`, so the two maps can never desync on disk). Staleness mirrors BM25 (`doc_count` + index meta `indexed_at` must both match) with one tightening: `indexed_at` must be **truthy** on both sides — a sidecar written without a real index meta proves nothing about its docstore state (BM25's None == None would pass), so persist is skipped when the meta has no stamp (also keeps docstore-only unit tests write-free) and load rejects it. Persist is skipped while an indexing run is active (same churn rationale as BM25); any load failure deletes the file and rebuilds — deleting it by hand is always safe. Removed by `_invalidate_retrieval_caches` alongside the BM25 sidecar.
- **Retune exclusivity (2026-07-04, item 2.7).** The shared BM25 retriever's `similarity_top_k` and the reranker's `top_n` are written in exactly ONE place: `rag/engine.py::_build_retrieval_pipeline`, which every consumer executes inside the same `_index_mutation_lock` hold that runs the retrieval/rerank — so tune and use are atomic w.r.t. concurrent requests. The manager's fetch paths (`_get_bm25_retriever` fast + double-checked branches, `_get_reranker` cached branch) are **read-only by contract** — the old fetch-time retunes ran under no lock / the load lock and let a deck run's `vault_search` (k=12) retrim a concurrent chat's in-flight pass (k=8). Adding a retune anywhere outside the locked pipeline build reintroduces the race. Pinned by `TestSharedRetunerRace`.
- `ObsidianVaultManager._get_reranker(model_name, top_n)` lazy-loads the cross-encoder on first chat that asks for reranking and caches the loaded model in memory. The cache and sticky-failure keys are `"{model}::{device_mode}"` where the mode comes from `vault_reranker_device` (`auto`/`cpu`/`mps`; unknown ⇒ `auto`, read via stat-cached `load_config()` on every call). `auto` constructs **without** a device argument — byte-identical to the pre-knob behaviour, so llama-index's `infer_torch_device()` picks MPS on Apple Silicon when available. Non-CPU modes run a one-pair warm-up `predict` at load time because Metal failures often surface on the first forward pass, not at construction (the engine's postprocessor walk does not degrade); any non-CPU failure retries on CPU and caches that instance for the session (warning emitted) instead of setting the sticky flag. A sticky `_reranker_failed` flag prevents re-attempting the same model+device after a CPU-path failure (the user must change `vault_reranker_model` or `vault_reranker_device` to retry — a deliberate signal that they want another attempt). The reranker is index-independent and survives `cleanup()` / `/api/reset` so a vault switch does not re-download weights.
- `llama-index-retrievers-bm25>=0.7,<0.8` powers hybrid retrieval and is compatible with the LlamaIndex 0.14.x core line. The BM25 retriever is built lazily in memory from the existing saved docstore on the first hybrid query and cached by chunk count, so enabling or updating BM25 does not require a vault re-index. `llama-index-postprocessor-sbert-rerank` + `sentence-transformers` power the cross-encoder. Both integrations are import-guarded in `rag/engine.py` (`BM25Retriever` / `SentenceTransformerRerank` resolve to `None` when their package is absent). The manager treats `None` as "feature unavailable" and falls back to the dense-only path; a missing-model warning surfaces once via `_emit` from `_get_reranker` (BM25 build failure is logged but does not warn the user since the dense leg still works).
- `launch.py::_maybe_enable_hf_offline()` sets `HF_HUB_OFFLINE=1` **before** the app imports when the configured reranker model already exists in the HF cache (`~/.cache/huggingface/hub/models--<repo>`), so sentence-transformers' ~25 per-load HEAD/GET requests to huggingface.co are skipped. It must run before `from app import app` because huggingface_hub reads the env var into a module constant at import time. First-run downloads still work (the env is left unset when the cache is empty); changing `vault_reranker_model` to an UNCACHED model mid-session cannot download until the next launch — the reranker-failure warning in `_get_reranker` tells the user to restart.
