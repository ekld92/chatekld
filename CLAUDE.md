# ChatEKLD 2026 Project Notes

ChatEKLD is a local macOS app for PDF summarisation and Obsidian vault RAG. It uses Flask, PyWebView, SQLite, LlamaIndex, Ollama, and LM Studio.

> **Layered docs.** This file holds the cross-cutting rules and invariants only. Deep implementation notes live in subtree `CLAUDE.md` files that load on demand when you work in those directories: `rag/CLAUDE.md` (indexing pipeline, locks, prewarm, retrieval mechanics), `core/llm/CLAUDE.md` (adapters, fallback policy, usage/pricing, tool wire formats), `core/agent/CLAUDE.md` (agent loop), `audit/CLAUDE.md` (Library Audit internals + endpoint details). History lives in `CHANGELOG.md`. Deferred items are under **Known Issues** at the bottom.

## Current Workflows

- Single Paper. Upload one PDF. Extract text. Stream a tunable summary with prompt presets, document-type prompts, audience, focus question, and generation controls.
- Obsidian Vault. Index `.md` and `.pdf` files. Query the vector index through dense retrieval, optional BM25 lexical retrieval, RRF fusion, and optional cross-encoder reranking; inspect the indexed-material manifest.
- Library Audit. Read-only reconciliation of the Obsidian vault against Zotero (SQLite snapshot + Better BibTeX `_master.bib`), local PDFs (annotations + duplicates) and macOS Finder tags. Six reports plus a per-citation-key inventory. Engine vendored from kb_harmonizer under `audit/`. Strictly manual — the scan runs only when the user clicks Run Scan.
- Deck Generator. Turn a topic + free-form instructions into a LaTeX **Beamer** deck grounded in the indexed vault, in the user's own template's house style. Pick (and edit in-app) a Beamer template; deckgen reuses its preamble + custom macros (`\citefoot`, `\commonlogo`), maps cites to the template's `_master.bib` when possible, and scaffolds a compile-ready `<slug>/<slug>.tex` + `Makefile` into the suite. **Emit-only by default** — the user compiles with `make`; the one exception is the opt-in, confirm-gated **Compile & Auto-Fix** loop (`/api/deck/compile-fix`): our own bounded `latexmk -no-shell-escape` run + LLM log-repair passes, every written repair screened and verified by a follow-up compile, `.bak` before the first write. Details in `deckgen/README.md`.
- Plain Chat. A RAG-free, multi-turn conversation with the globally-configured provider/model — no vault retrieval, no agent loop, no tools. Stateless server: the browser owns the `{role,content}` history and sends the last 20 turns on every send; conversation is ephemeral (lost on reload). Two persisted knobs (`chat_temperature`/`chat_system_prompt`); the system prompt is the **full** system prompt (no retrieval grounding to protect). Streams through the unified `core.llm.chat.stream_chat_messages` for both local and online providers, falling back only before the first token.

- `api/security.py` owns local-origin request checks and `sanitise_error_msg`.
- `api/validators.py` owns request-body coercion helpers (`coerce_int_in_range`, `coerce_float_in_range`, `coerce_bool`, `coerce_enum`, `coerce_regex`, `coerce_non_empty_string`, `coerce_string_max_len`, `first_valid`, `MISSING`). Every route layer imports from here rather than reimplementing NaN/Inf/range/clamp logic — `api/routes/vault.py::_resolve_chat_params` is the canonical caller.
- `api/sse.py` owns the shared backend SSE skeleton (item 4.3): `run_sse_worker(worker, *, consumer_timeout_s, preflight_msgs, release)` — bounded queue + daemon worker + stall guard + `data:`-framed drain + `Cache-Control: no-cache`/`X-Accel-Buffering: no` — and `_SSEOpGuard` (the promoted `_DeckOpGuard`: *release* fires exactly once, when the stream has ended — generator finally OR `call_on_close` — AND any spawned worker exited; pinned by `test_deck.py::TestDeckOpGuard`). All five streaming routes (vault chat, plain chat, refactor plan, deck, paper summarise) run through it. `sanitise_error_msg` is imported from `api.security` (its owner) — the first Batch 2 cut imported it from `core.utils` and broke app boot.
- `core/paths.py` owns `resolve_under_root(raw, root, *, must_exist, must_be_dir, must_be_file, exts, deny_root, max_len)` (item 4.2) — the traversal/NUL/escape-rejecting validator for path inputs that resolve **under a known root**; returns the canonical posix rel or `None`. It is the single chokepoint behind **all** of `refactor.py`'s route validators, and behind `deck.py`'s figure-copy + `audit.py`'s scan-pdf/reveal-root resolution. Path inputs that are user-chosen **absolute** locations with no under-root anchor — `deck.py`'s `_resolve_template_path`/`_resolve_out_dir`/`_resolve_existing_deck` and `audit.py`'s `_normalise_rel_path`/`_normalise_abs_path` — cannot use it (it needs a root and returns a rel) and each apply their own inline NUL/newline + `realpath` + `_DENY_PREFIXES` guards, so a future hardening of `resolve_under_root` must be mirrored into those hand-rolled validators. Scope-locked wrappers (e.g. `refactor.py::_resolve_scope_note_rel`) resolve against the VAULT root and prefix-check the scope rel — resolving against the scope dir instead doubles the scope segment (the repaired first-cut bug, pinned by `test_scope_note_rel_validator_rejects_escapes`).

Its **core is app-independent and `requests`-free**: `template.py` (split a template into preamble/opening/closing, scan custom macros by following `\usepackage` into local `.sty`, parse the `\addbibresource` bib, find the suite root; **parse caches (5.2):** the per-`.bib` key index and per-`.sty` capped text are cached keyed on `(path, size, mtime_ns)` — thesaurus-cache pattern, self-invalidating on edit, stdlib-only — so repeat generate/augment/compile calls stop re-reading a multi-MB suite bib), `outline.py`, `sections.py`, `prompts.py` (now also the **augment** system prompt + message builder — `augment_system_prompt`/`build_augment_message`, per-operation deepen｜table｜new_section), `assemble.py` (`assemble_with_template` + `sanitize_section` + `validate`'s dangerous-macro + hallucinated-citekey guards), `augment.py` (the **augment core**, pure: `split_deck` decomposes a *finished* deck into preamble/opening/`\section` spans/closing reusing `template`'s boundary detection, and `replace_section`/`replace_region`/`insert_section` splice an edit back **byte-identically outside the edited span** — the invariant `replace_section(tex, sec, sec.body) == tex` is what lets a section-scoped edit ride the whole-document sha256 guard), `scaffold.py` (`scaffold_deck` writes `<slug>/<slug>.tex` + `Makefile`; `write_deck_tex` rewrites only a scaffolded `.tex`; `write_deck_at` atomically overwrites an *arbitrary* validated deck `.tex` for augment-apply; keeps its own atomic-writer), `review.py` (integrity-review prompt/parse/`screen_repair` — reused by augment as the safety gate), `retry.py` (`chat_with_retry` — pure, cancel-aware per-turn retry wrapping `client.chat`; used by `outline.py`/`sections.py` for the deck route's per-section resilience), `checkpoint.py` (pure per-section checkpoint/resume: `compute_job_key` over the content-determining inputs, atomic `load`/`save`/`delete`/`prune` of a manifest holding the outline + each completed `SectionOutput` under `BASE_DIR/deckgen/checkpoints/`), `result.py` (`ChatResult`, shared, no third-party imports). `client.py` is the HTTP driver for the **CLI** (`__main__.py`); `inprocess.py` is the **only** app-coupled module — `InProcessChatRunner` drives `core.agent.run_agent_loop` over `rag.vault.obsidian_manager` directly, exposing the same `.chat(...) -> ChatResult` contract the CLI client does. Imports flow deckgen-core ← {client, inprocess}; the core never imports the app. **Augment is in-app only** (no CLI flag).
- `api/routes/deck.py` owns the `/api/deck/*` blueprint: `load-template` (validate + scan), `generate` (SSE, in-process orchestration), the optional final-stage `.tex` integrity review + auto-repair, `apply-repair` (write a confirmed repair over the on-disk deck), and the native file/folder pickers. Path validators reject traversal/system roots, mirroring `audit.py`. **Per-section resilience** (`generate`): each outline/section turn is retried in-process up to `deck_section_max_attempts` (1–5, default 3) times with linear `deck_retry_backoff_s` (0–30, default 3) backoff via `deckgen.retry.chat_with_retry` (cancel-aware, surfaces a `{"info"}` per retry); a section that still fails degrades to `generate_section`'s placeholder frame. Crucially the worker's `_on_event` **relabels a turn-level `{"error"}` (a per-section provider failure the agent loop emits) to a non-fatal `{"info"}`**, because the shared SSE consumer (`api/sse.py::run_sse_worker`) treats any `{"error"}` frame as terminal — without the relabel one weak section would discard the whole deck (the historical random-failure mode). Only the worker's explicit `put({"error"})` (an unrecoverable outline failure) stays fatal; SDK-level retries are off (`get_lmstudio_client` `max_retries=0`) so this in-process retry is the single, bounded, user-visible recovery layer. **Per-section checkpoint/resume** (`deck_resume_enabled`, default on; body `force_fresh` discards): the worker persists the parsed outline and each completed section to `BASE_DIR/deckgen/checkpoints/<job_key>.json` (`deckgen.checkpoint`, atomic, keyed by `compute_job_key` over the content-determining inputs — not the sampling/retry knobs) **immediately after each section**, so a re-submitted identical request reuses the saved outline (skips the outline turn) + every already-generated section and resumes from the first missing one; a fully successful scaffold deletes the checkpoint (+ `prune` to 10), a scaffold failure keeps it for the next run. **A placeholder section (every retry failed) is deliberately NOT checkpointed and is never reused on resume** — both the persist and the reuse paths guard on `not out.placeholder`, so resume retries a transiently-failed section rather than freezing the failure in (it still rides in the *current* run's `section_outputs` so that run's deck assembles). The terminal deck frame carries `resumed`/`reused_sections` (counting only genuinely-reused real sections, not placeholders). The integrity review is **opt-in** (`deck_review_enabled` default off, body field `review_enabled`): when on, after assemble+validate and before scaffold it makes **one RAG-free `core.llm.chat.stream_chat_messages` call** (`_run_integrity_review` — the one place `deck.py` reaches into `core.llm`, parallel to `inprocess.py`→`core.agent`) over the whole assembled deck and screens any proposed repair through `deckgen.review.screen_repair`. The repair is only ever **offered** (carried in the terminal `{"deck": {…, tex_sha256, "review": {ran, issues, changed, repaired_tex, repaired_warnings, truncated, repair_truncated, error}}}` frame), never auto-written — the user applies it via `POST /api/deck/apply-repair` (`confirm: true`; requires the generate frame's `tex_sha256` as `base_sha256`, a stale-diff guard that **409s** if the on-disk deck changed since the review; **re-screens** the submitted `.tex` via `screen_repair` against the current deck so a smuggled macro can't ride in through the client body; copies `<slug>.tex.bak` **before** overwriting (item 2.9 — backup parity with apply-augment/compile-fix; a failed backup aborts), rewrites **only** `<slug>.tex` via `scaffold.write_deck_tex`, never the `Makefile`). `deck_review_model` ("" ⇒ chat model) and `deck_review_max_tokens` (256–16384, must fit the re-emitted document) are Settings-window/`/api/config`-set, never body overrides. `deckgen/review.py` (app-independent, LLM/transport-free, like `assemble.py`/`scaffold.py`) is the prompt/parse/screen core: `build_review_messages`, `parse_review` (issue bullets + the last whole-document fenced block; a truncated/unclosed block degrades to issues-only and sets `repair_truncated`; the "no issues" test is an **anchored whole-body** match, not a substring, so prose containing "none" isn't silently dropped), and `screen_repair` (the safety gate — **refuses** a repair that introduces a dangerous compile-time macro the original lacked, or loses the single `document` env / all frames; else returns the repaired `.tex` + residual structural `validate()` warnings). Dangerous-macro detection is the shared `assemble.find_dangerous_macros` — it catches the `\csname write18\endcsname` / `\@@input` obfuscations the bare `\macro` regex missed (residual catcode/char-by-char limits documented at the function). So deckgen's "`inprocess.py` is the only app-coupled module" invariant holds for the **core** (the model call lives in the route). Both SSE routes (`generate`, `augment`) run through the shared backend SSE skeleton `api/sse.py::run_sse_worker` (queue + daemon worker + stall-timeout + thread-leak guard; promoted from deck's `_run_deck_sse` by item 4.3 — all five streaming routes now use it) and hold a process-wide `_DECK_OP_LOCK` so **one deck operation runs at a time** (a concurrent request gets **409**). **Lock-release ownership (2026-07-04, item 2.2):** release is owned by the skeleton's `_SSEOpGuard` (né `_DeckOpGuard`), firing exactly once when the stream has ended (generator finally **or** the Response's `call_on_close` — the latter covers a generator closed before first iteration, which previously leaked the lock for the process lifetime) **and** any spawned worker thread has exited — so a consumer stall/disconnect can no longer release the lock while the worker is still writing (the zombie-writer race; a truly wedged worker now keeps new deck ops 409ing instead of letting them clobber, and the worker's own per-turn/compile/review bounds are what end it). The generate worker additionally cancel-gates before assemble/figure-copy and before scaffold, and the augment worker before its staging write, so an abandoned run stops writing (the per-section checkpoint makes the skipped scaffold resumable). Pinned by `test_deck.py::TestDeckOpGuard` (now importing from `api.sse`). The two **synchronous writers** `apply-repair`/`apply-augment` also take `_DECK_OP_LOCK` (non-blocking → **409** if busy, released in a `finally`) across their read→screen→backup→write, so the "one deck operation at a time" invariant covers writers too — an apply can't race a streaming generate/augment or another apply and clobber a deck mid-flight. On-disk stale-diff tokens are raw-byte sha (`_file_sha256`); decks that may be overwritten are read via `_read_deck_strict` (raw-byte sha + **UTF-8-only**, **rejects** non-UTF-8 / >1 MB rather than lossily decoding or truncating — also fixed apply-repair's `UnicodeDecodeError`-as-500). **Augment endpoints** (deepen/extend an *existing* deck — deckgen-made or hand-written; **overwrites a user file**, so stricter than apply-repair, matching the refactor write model): `deck-sections` (read-only — `split_deck` → `{sections:[{index,title}], deck_sha256}` for the scope picker), `augment` (SSE, one in-process agent turn over the targeted section / whole section-region / a new-section insert; `operation`/`scope` 400 on a present-but-invalid value, section-index/zero-section/**truncation** guards in the worker — an over-cap whole-deck *deepen/table* is refused so a clipped-view replace can't silently drop slides; a turn-level `{"error"}` the agent loop emits during the augment turn is **relabelled to a non-fatal `{"info"}`** by the worker's `_on_event` (parity with generate, 2026-07-05 audit m5 — a single transient provider blip no longer discards the whole augmentation; the empty-body guard still terminates cleanly if the run truly produced nothing); then `sanitize_section` → `augment.replace_section`/`replace_region`/`insert_section` → `validate` + `screen_repair(original, proposed)` with **de-duplicated** warnings; on success **stages** the screened proposal server-side under `BASE_DIR/deckgen/staging/` (the preview's figure resolution runs `copy=False` — the vault→`figures/` copies ride the staging record as `pending_figures` and are performed by apply-augment after the backup, so the preview is genuinely write-free, item 2.9); terminal `{"augment": {proposed_tex, changed, warnings, rejected_reason, deck_path, deck_sha256, counts{frames/sections before→after}, operation, scope, section_index}}` frame — **writes nothing to the deck**), `apply-augment` (`{deck_path, base_sha256, confirm:true}` — **no client `tex`**: reads the proposal from staging, requires on-disk == client base **and** staged base (else **409**), **re-screens** via `screen_repair`, copies `<deck>.tex.bak` **before** overwriting (aborts if the backup fails), writes via `scaffold.write_deck_at`, clears the stage). The free-text instruction is the *trusted* task; the deck content is *untrusted* `<existing>` source (parallel to refactor `custom_edit`). `_resolve_existing_deck` validates an absolute `.tex` not under `_DENY_PREFIXES` (it can write back, unlike the read-only `_resolve_template_path`). No diagrams/TikZ — text frames + `tabular` only; **no journal** (the `.tex.bak` is the recovery path). `static/js/deck.js` owns the augment panel (`window.deckPickAugmentDeck`/`deckLoadDeckSections`/`deckAugScopeChanged`/`deckAugmentPreview`, `ui.js`+`api.js`-only; a frame/section count-delta banner flags any decrease, and Generate + Augment share a busy state so only one runs).
- `refactor/` is the Note Refactor analyzer package (**Phase 1 read-only + Phase 1.5 hub**, mirrors the `deckgen/` core/route split): `result.py` (third-party-free dataclasses + `*_frame()`/`to_jsonable()` serializers; ## Provider Rules

- `provider` selects `ollama`, `lm_studio`, `openai`, `anthropic`, or `google`.
- Online providers (`openai`, `anthropic`, `google`) are CHAT-ONLY — they never expose an embedding interface. When the active chat provider is online, the indexer, vault chat, and prewarm resolve a local embed provider via `core.config.resolve_embed_provider(cfg, provider_name)` (falls back to `embed_provider`, default `ollama`).
- `core.providers.get_provider(name)` always returns a local `Provider`; called with an online name it silently substitutes the configured local embed provider. `core.llm.get_llm_provider(name)` returns the unified `LLMProvider` chat interface for both local and online.
- API keys come from env vars (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`) at call time — never persisted to `config.json`, never returned by `/api/config`. `api.security.sanitise_error_msg` runs through `core.llm.redact.redact` so an accidental key leak in an exception is stripped before the UI sees it.
- Per-provider model selections persist independently: `llm` for local providers, `openai_model` / `anthropic_model` / `google_model` for online; `core.config.resolve_chat_model(cfg, provider_name)` picks the right field. `/api/config` routes the UI's generic `llm` field into the active online provider's key and then **drops** `llm` from the payload — persisting an online model name into `llm` would clobber the local selection (pinned by `smoke_test.py::test_03b`).
- `/api/models` routes to the active provider. For online providers it returns `CURATED_MODELS` merged with a best-effort, key-gated, short-TTL-cached live fetch from the provider's models endpoint (`core.llm.model_listing`); curated stays first and degrades to curated-only without a key / offline / on error. `/api/status` ok=True for online providers just means the API key is set. `/api/pull` is Ollama-only (chunks serialised via `.model_dump()`).
- Fallback policy: `fallback_provider` + `fallback_on` config keys; falls back only on transient errors and only **before the first streamed token**; hard quota/billing errors are terminal. Full mechanics in `core/llm/CLAUDE.md`.
- Token / USD-cost usage is recorded by `core.llm.usage.usage_tracker` for every request and surfaced on `/api/usage`; pricing in `PRICING_TABLE`, overridable via `llm_pricing_overrides`. Details in `core/llm/CLAUDE.md`.
- OCR and vision settings are separate from chat provider settings. `/api/vision-models?provider=...` lists models for Ollama or LM Studio only; `ocr_provider` / `vision_provider` decide which backend receives image calls.

## Vault Rules

- The indexer reads `.md` and `.pdf` files. `.docx` is excluded by design. PDFs are folder-driven (every `.pdf` in an included folder is eligible). Images are note-driven: only files whose extension appears in `vault_image_exts` AND that are referenced from an included markdown note are described by the vision model and embedded; `[]` disables image indexing; skipped images are counted in `skipped_image_count`.
- `obsidian_vault_path` is validated before save — broad roots and system directories are rejected. `vault_exclude_dirs` stores vault-relative paths; exclusions are applied before files are read.
- Index updates are incremental: unchanged chunks are skipped by document hash, changed chunks are replaced, stale chunks are deleted. Indexing streams end-to-end (peak RAM bounded by the largest single document); checkpoints persist after 500 inserts *and* ≥ 10 minutes since the previous one (each checkpoint rewrites the whole store); per-insert failures are tolerated up to 20 consecutive. Details in `rag/CLAUDE.md`.
- **Vector store backend** (`vault_vector_backend`, `simple` | `lancedb`, default `simple`). `simple` is the legacy JSON `SimpleVectorStore`; `lancedb` is the binary Apache-Arrow backend (low resident RAM, fast cold start, transactional inserts — no full-store JSON dump for *vectors*, though the docstore/index-store JSON still persist at the checkpoint cadence). The knob governs only **fresh** builds; an existing index's backend is authoritative from `obsidian_meta.json` (`vector_backend`; missing ⇒ `simple`). Migrate an existing vault in place with **no re-embedding** via `python scripts/migrate_vector_store.py` (app closed), or set the knob and reindex. On lancedb the embeddings are unit-normalized so default-L2 ranking equals cosine; list metadata (`attachments`) is JSON-stringified in the row while the docstore keeps the list (`store_nodes_override=True`). Each streaming insert is its own LanceDB transaction (a version manifest that re-lists every fragment, so an unbounded run grows `_versions/` ~O(n²)); the indexer compacts + prunes superseded versions (`Table.optimize`) at each checkpoint **and** every `_LANCEDB_COMPACT_EVERY` (2000) inserts, keeping the on-disk footprint O(n). `scripts/compact_lancedb.py` is a one-off offline reclaim for an index bloated before this existed. Mechanics in `rag/CLAUDE.md`.
- Chunking is **pinned** (changing it forces a reindex; `TestReindexInvariant` guards it): `.md` at heading boundaries (`MarkdownNodeParser`), then a **conditional secondary cap pass** that sub-splits only sections over `MD_MAX_CHUNK_TOKENS` (1024, `core/constants.py` — a pinned constant, not a config knob, since it changes chunk IDs) so a long single-heading section is no longer silently truncated by the embedding model's ~2048-token limit; PDF flat-text via `SentenceSplitter` (512-token chunks, 64-token overlap). Chunk IDs are `"{rel_path}::{sha1(f"{i}\n{text}")[:16]}"`; page-range documents from large PDFs (> 1000 pages, one document per 1000-page range) salt the hash input with their `page_start` (the input becomes `f"{page_start}:{i}\n{text}"`) so ranges of the same file cannot collide — single-document files keep the unsalted input, so they never re-embed. The secondary pass is churn-free for under-cap sections (it keeps the original node object, so `i`+text and the doc_id are byte-identical); only notes containing an oversized section re-chunk that section onward. Details in `rag/CLAUDE.md`.
- Vault RAG prompts treat retrieved notes/PDFs as untrusted source text, apply a similarity cutoff, and construct per-query LLM/embedding objects instead of mutating global LlamaIndex settings.
- Vault chat accepts query-time knobs as per-request overrides on `/api/obsidian/chat`, resolved by `_resolve_chat_params` (body wins, then the persisted `vault_*` config key, then the engine default) — **all reindex-free**:
  - Generation: `top_k` (1-32, default 8), `similarity_cutoff` (0.0-1.0, default 0.25), `prompt_mode` (`strict`/`balanced`/`exploratory`/`concise`), `temperature` (0.0-2.0, default 0.3), `system_prompt` (≤ `SYSTEM_PROMPT_LIMIT` = 4000 chars; a prefix over the mode template — the safety preamble stays app-controlled).
  - Retrieval: `hybrid_enabled`, `reranker_enabled` (both default `True`; **config-only, not body overrides:** `vault_rrf_dense_weight` / `vault_rrf_bm25_weight` (0.0–10.0, default 1.0) weight the fusion legs — at the 1.0 defaults fusion stays `reciprocal_rerank` byte-identically, any other value switches that query to `relative_score`, the only LlamaIndex mode that applies `retriever_weights`; score scales differ between modes, so tune against `tests/eval/` — see `rag/CLAUDE.md`), `mmr_enabled` + `mmr_lambda` (default off / 0.5), `query_expansion` + `num_queries` (default off / 3, local providers only), `rerank_pool_ceiling` (10-200, default 50), `wikilink_expansion` (default off; the only wikilink knob that is a body override).
  - Vault thesaurus (curated glossary files; default paths `_abreviations.md` / `_tags.md` at the vault root): `thesaurus_expansion` (default off; body override) drives deterministic synonym query expansion via `_ThesaurusExpansionRetriever`; `primer_enabled` (default off; body override) injects a small app-controlled glossary block into the system prompt so the LLM can read the vault's shorthand. **Config-only** (not body overrides): the file paths `vault_thesaurus_abbrev_path` / `vault_thesaurus_tags_path` (vault-relative, "" disables the slot), the depth knobs `vault_thesaurus_max_variants` (1-8, default 3) and `vault_primer_max_chars` (500-8000, default 1500), and the content overrides `vault_primer_header` / `vault_primer_core_terms` ("" ⇒ built-in defaults). Both features no-op when neither configured file exists. See the **Vault thesaurus** bullet below and `rag/CLAUDE.md`.
- **Wikilink graph expansion** (`vault_wikilink_expansion`, default `False`; body field `wikilink_expansion`). Query-time, **reindex-free**: after retrieval, the result set is widened with chunks from notes the seeds link to **and** notes that link into them (outbound ∪ backlinks), fetched from the docstore, *before* the rerank stage decides what survives (rerank-gated — irrelevant neighbours drop out, final answer size unchanged). **Only takes effect when the reranker is active** (the default): the reranker (`top_n = top_k`) is what trims seeds+neighbours back to `top_k`, so with the reranker off the engine skips the wrap (and `stream_chat`/`retrieve` skip the graph build) rather than push >`top_k` chunks at the model. The note→note graph is built lazily from the docstore (regex over `.md` node text + Obsidian shortest-path resolution against indexed sources) and cached like BM25 — invalidated by `_invalidate_retrieval_caches`, optionally warmed by `prewarm`, and persisted to an on-disk sidecar (`obsidian_storage/wikilink_sidecar.json`, BM25-sidecar discipline; mechanics in `rag/CLAUDE.md`) so a process-cold chat loads it instead of re-sweeping the docstore; `_get_wikilink_index`/`_WikilinkGraph` in `rag/vault.py`, `_WikilinkExpansionRetriever` (one wrap in `_build_retrieval_pipeline`, before the postprocessors) in `rag/engine.py`. Caps are **config-only** (`vault_wikilink_neighbor_cap` 1-100/10, `vault_wikilink_node_cap` 1-200/24, `vault_wikilink_score_decay` 0.0-1.0/0.5; the engine reads them from `cfg` like the rerank-pool knobs). Applies to single-shot RAG (and the agent's RAG fallback) only — **not** the agent's active `vault_search`, matching `mmr_enabled`/`query_expansion`. The UI toggle (`#vault-wikilink-enabled`) is owned by `vault.js` like the other live retrieval knobs.
- **Vault thesaurus** (`vault_thesaurus_expansion` + `vault_primer_enabled`, both default `False`; body fields `thesaurus_expansion` / `primer_enabled`). Query-time, **reindex-free** and **embed-free**. The thesaurus is parsed from two hand-curated Markdown tables whose **vault-relative paths are configurable** — `vault_thesaurus_abbrev_path` (default `_abreviations.md`) and `vault_thesaurus_tags_path` (default `_tags.md`); an empty value disables that slot. `_get_thesaurus()` resolves each under the vault root (`_resolve_thesaurus_rel` rejects traversal/absolute/escaping, defence-in-depth over the `/api/config` shape check), reads + parses them lazily (cached by the resolved paths **plus** each file's `(size, mtime_ns)` signature, so an edit to a file *or* a config path change refreshes on the next chat; cleared on vault switch / `cleanup()`), and returns `None` — disabling both features — when neither resolved file exists. The app-independent parser/consumers live in `rag/thesaurus.py` (stdlib-only, like `rag/lancedb_store.py`): (1) **query expansion** (`expand_query`) substitutes a matched concept/abbreviation with each known synonym, one rewritten query variant per synonym (config-capped at `vault_thesaurus_max_variants`), each retrieved separately and rank-fused via `_ThesaurusExpansionRetriever` in `rag/engine.py` — so the dense leg is not diluted and BM25 can hit the synonym token; (2) **system-prompt primer** (`build_primer`) renders a compact glossary block (≤ `vault_primer_max_chars`, query-relevant rows first) prepended to the answer-mode template via `_apply_vault_primer`, so the LLM can *read* shorthand that survives into the retrieved context. Both apply to single-shot RAG (and the agent's RAG fallback) only — **not** the agent's active `vault_search`, matching `wikilink_expansion`/`mmr_enabled`/`query_expansion`. **Domain defaults (overridable):** `rag/thesaurus.py`'s built-ins are tuned for the maintainer's bilingual French/English psychiatry/clinical-research vault, but the corpus-specific pieces are now config-driven — `vault_primer_header` replaces the `_PRIMER_HEADER` intro sentence and `vault_primer_core_terms` (comma-separated) replaces the `_CORE_PRIMER_TERMS` priority list (both empty ⇒ built-ins), and the parser accepts **English** column headers (`Abbreviation | Meaning`, `Description | Tag`) as well as French. The `≠` collision convention stays a documented marker. Defaults-off so it is invisible to vaults that don't opt in. Covered by `test_thesaurus.py`.
- Reranker model is set via the `vault_reranker_model` config key (editable in the **LLM Settings** window, or `POST /api/config` directly — both local-only) but is **never** a `/api/obsidian/chat` body override (prevents a malicious page from triggering arbitrary HuggingFace downloads). `launch.py` enables `HF_HUB_OFFLINE=1` at startup when the model is already cached. `vault_reranker_device` (`auto`/`cpu`/`mps`, default `auto`, also Settings-window/`/api/config`-set, not a body override) selects the cross-encoder's device: `auto` keeps the library's own inference (MPS on Apple Silicon when available), `cpu` is the escape hatch that keeps unified memory free for the LLM, and any non-CPU failure degrades to CPU for the session instead of disabling reranking.
- `vault_prewarm_enabled` (default `True`) gates the launch-time prewarm inside `ObsidianVaultManager.prewarm()`: when off, prewarm reports `skipped` (terminal for the UI — Send stays enabled) and the first chat lazy-loads instead. Defers the multi-GB load; does not remove it.
- Retrieval mechanics (RRF fusion, MMR, rerank pool sizing, cutoff semantics, BM25/reranker caches, the persisted BM25 sidecar under `obsidian_storage/bm25_index/`, prewarm, lock discipline) are documented in `rag/CLAUDE.md`.

## Agent Mode (Vault Chat)

- Opt-in alternative to single-shot RAG on `/api/obsidian/chat`: `#vault-agent-enabled` checkbox (`vault_agent_enabled`, default `False`) or body field `agent_enabled`. When off, `stream_chat` runs byte-for-byte identically — the agent layer is entirely additive.
- Three tools over `ObsidianVaultManager`: `vault_search(query, top_k≤12)` (results carry `is_image` from the chunk metadata the indexer stamps on vision-described images — the deck runner's figure prompt keys on it), `vault_read_note(rel_path)`, `vault_list_materials(filter?, limit≤200)`. Tool output is truncated and wrapped in the untrusted-content guard (per-call nonce'd tag — content containing a literal closing tag can't escape the wrapper).
- **Tool names must match `^[A-Za-z_][A-Za-z0-9_-]{0,63}$`** — `ToolRegistry` raises `ValueError` at construction otherwise. This is the strictest intersection of the providers' rules; the pre-2026-07-02 dotted names (`vault.search`) were 400'd by OpenAI *and* Anthropic on every tool-enabled request (the mocked test suite can't see provider-side validation — `test_llm.py::TestProviderToolPayloadContract` is the in-process stand-in).
- Loop budget `vault_agent_max_iterations` (default 6, clamped 1–12); 300 s wall-clock cap. Two consecutive malformed tool-call iterations fall back to plain RAG against the original message; a session-level capability warning fires after two consecutive fallback turns.
- Full loop/fallback/wire-format details in `core/agent/CLAUDE.md` and `core/llm/CLAUDE.md`.

## OCR / Vision Availability Caching

The configured OCR/Vision model is always the model that gets called. `check_availability()` is informational only (cached 60 s); it never gates a call and never rewrites `self.model`. A negative-result cooldown fast-fails calls after a failure so per-page traffic cannot hammer a missing model — the window is the `vision_failure_cooldown_s` config key (0–600, default 30; **0 disables** — every image retries immediately), read per call via `services.vision._failure_cooldown_s` (its own reader, NOT `_cfg_bounded_int`, whose `≤0 ⇒ default` contract would erase the disable). During indexing, images inside the window are silently skipped, so a shorter window trades retry traffic against fewer dropped images after a transient blip. `set_model()` / `set_provider()` invalidate both caches immediately.

**Call bounds (always on).** Every `VisionManager.describe_image` / `GLMOCRManager.extract_page_text` call is bounded so a runaway or stuck local model cannot stall a multi-hour indexing run — the OpenAI/LM Studio SDK default is otherwise a 600 s timeout × 2 retries (= 30 min per stuck image). The transports (`_chat_lm_studio_image`, `_chat_ollama_image`) take keyword-only `timeout`/`max_tokens`: LM Studio passes `timeout` + `max_retries=VISION_MAX_RETRIES` (0) to `openai.OpenAI(...)` and `max_tokens` to `create(...)`; Ollama uses the timed `_ollama_client(OLLAMA_HOST, timeout)` and `options={"num_predict": max_tokens}`. The timeout/cap come from `vision_timeout_s` (5-600, default 120; **always on — no "0 = off"** unlike `local_request_timeout_s`), `vision_max_tokens` (default 1536), and `ocr_max_tokens` (default 4096), read **per call** via `_cfg_bounded_int` (lazy `load_config`, like `local_request_timeout()`) so a Settings change applies without restart; a missing/≤0/garbage value falls back to the hard `DEFAULT_*` constant and a positive out-of-range value is clamped to the validator's range (re-asserting the bound for a hand-edited `config.json`). The **description path** (not OCR — that would hurt text legibility) additionally pre-emptively downscales images whose longest side exceeds `VISION_IMAGE_MAX_SIDE` (1568 px) via `_fit_base64_image_to_max_side`, a best-effort PNG re-encode that returns the original on any failure (undecodable image — a format PIL can't open, or a corrupt file; note HEIC **is** decodable here, since `pillow-heif` is a pinned dependency and `services/vision.py` calls `register_heif_opener()` at import — or non-image input). Downscale is invisible to the description cache (`rag/vault.py` keys it on the *original* image bytes); the OCR in-memory cache likewise keys on the *original* `base64_png`, not the overflow-downscaled payload. `_chat_lm_studio_image` never forwards `timeout=None` (the SDK can read that as "no timeout").

## JS Module Hierarchy

`api.js` is the leaf module — it imports nothing from this project. `ui.js` imports only `api.js` (for `logError`). All other modules (`config.js`, `vault.js`, `summarizer.js`, `audit.js`, `deck.js`, `settings.js`, `plainchat.js`, `prompts.js`) import only from `ui.js` and `api.js`, with the documented exceptions that `vault.js` and `summarizer.js` also import from `config.js` (to read active settings and model choices without causing circular dependencies). `app.js` is the root and imports from all others. This prevents circular dependencies; in particular `config.js` must never import from `app.js`.

`ui.js::renderExampleChips(targetEl, examples, {mountEl?, title?})` is the shared click-to-insert "example prompt" primitive: it renders a row of chips that, on click, **fill an empty field but append (after a blank line) to a non-empty one** so a user's draft is never clobbered, then dispatch an `input` event (so debounced-save / `oninput` listeners react as if typed), pure-DOM via `textContent` (never innerHTML), idempotent per field. The French, vault-tuned example sets live at the call sites — `app.js::wireExamplePrompts()` for the three static fields (deck instructions, vault-chat system prompt, plain-chat input) and `refactor.js` for the per-note free-prompt textarea (created at render time, so it can't be wired statically).

`ui.js::confirmInline(anchorEl, {message, confirmLabel, onConfirm})` is the shared destructive-action ceremony (Track 6e): a one-shot inline confirm strip, Cancel first in DOM/focus order, replace-not-stack. **Every button that overwrites a user file or bulk-reverts vault state must route through it** — the deck apply-repair/apply-augment handlers and refactor's Revert-all do, pinned by `tests/js/confirmCeremony.test.js` (including a source ratchet on deck.js). Streamed generation announces its lifecycle through `ui.js::announceStatus` → the static `#sr-status-announcer` region (Track 6b; `tests/js/liveRegions.test.js` pins which status elements are static live regions — add `role="status"` in `index.html`, not at write time, for anything AT must hear). Theme color tokens are contrast-linted: `tests/js/contrastTokens.test.js` recomputes WCAG ratios from `styles.css` per theme (Track 6a) — when adding a token or reusing one on a new surface, add the (fg, bg, min) pair there; `--accent-hover` is text-grade and `--accent-hover-bg` is the hover surface (one value cannot be both).

Third-party JS is vendored under `static/js/vendor/` — currently `marked.min.js` (v15.0.12, MIT). Never load runtime JS from a CDN: the PyWebView renderer's network is independent of the Python process, and a failed CDN fetch used to surface as `ReferenceError: Can't find variable: marked`, destroying the chat answer at render time. `vault.js::_renderAnswer` falls back to plain text when `marked` is unavailable. User-controlled strings (folder names, paths) must never be interpolated into `innerHTML` — use `createElement`/`textContent` (see `renderExclusions`) or an escape helper (see `audit.js::_esc`).

409 `{shared:true}` when shared), `restore` (POST `{op_id}` or `{all:true}` → reverse via the manifest), `manifest` (GET → `{ops:[...]}` for the restore UI).
- `POST /api/plainchat` (Plain Chat) — body `{messages:[{role∈{user,assistant}, content}], system_prompt?, temperature?}`; RAG-free multi-turn SSE stream (`{info}`/`{token}`/`{error}` + `[DONE]`, no agent/iteration/tool frames). `messages` is capped to the last 20 turns + per-message content length; `system_prompt` capped at `SYSTEM_PROMPT_LIMIT`; `temperature` coerced 0.0–2.0. Provider/model resolved from config (`resolve_chat_model`).

Most routes require `X-Requested-With: ChatEKLD`.

## SSE Contract

Streaming routes emit JSON events, one per `data:` frame, ending with `data: [DONE]`:

```text
data: {"token": "..."}     # answer tokens
data: {"error": "..."}     # structured error (also mid-stream, before [DONE])
data: {"info": "..."}      # stage labels, fallback notices, agent usage footer
```

Single-shot `/api/obsidian/chat` additionally emits one `{"retrieval": [{source, score, is_image}, …]}` frame after retrieval and before the token stream (rendered as the collapsed "Retrieval Context" panel in `vault.js`); the deck routes' terminal frames are `{"deck": …}` / `{"augment": …}` / `{"compile": …}` as documented under API Routes.

Agent-mode `/api/obsidian/chat` adds four event types before the final token stream:

```text
data: {"iteration": N}
data: {"thought": "..."}
data: {"tool_call": {"id": "...", "name": "...", "arguments": {...}}}
data: {"tool_result": {"tool_call_id": "...", "content": "...", "is_error": false, "truncated": false}}
```

## Limits

- PDF upload limit: 500 MB. PDF extraction timeout: 600 seconds. PDF page ceiling: `PDF_MAX_PAGES` = 20 000 (shared by the upload worker and the vault indexer; both extract in 1000-page ranges, so the ceiling bounds time, not memory).
- System prompt limit: `SYSTEM_PROMPT_LIMIT` = 4000 characters (shared by single-paper and vault-chat `system_prompt`; defined in `core/constants.py`).
- Feedback field cap: 10 000 characters per string field.
- Obsidian operation lock TTL: 3600 seconds per acquisition.
- **Timeout chain** (nested; outer must exceed inner or the inner cap is defeated): `local_request_timeout_s` (per local HTTP call) ≤ `agent_wall_clock_s` (one agent turn / each deck section, the exact user cap) ≤ SSE consumer `event_q.get(timeout=max(cap, SSE_SINGLE_SHOT_FLOOR_S) + SSE_STALL_MARGIN_S)` ≤ frontend fetch abort. The cap is config-driven (`agent_wall_clock_s`, default 300, parsed defensively with `coerce_int_in_range` so a hand-edited config can't crash the route). The consumer stall guard and the vault.js abort **derive** from it, but both are **floored at `SSE_SINGLE_SHOT_FLOOR_S = 300`** (`SSE_STALL_MARGIN_S = 30`; vault.js abort = `(max(cap,300)+60)*1000`) — both constants now live in `core/constants.py`, shared by the vault, deck, and plain-chat SSE routes (plain chat has no agent cap, so its consumer wait is just `SSE_SINGLE_SHOT_FLOOR_S + SSE_STALL_MARGIN_S`): the agent path is bounded by its own deadline, but the *same* consumer loop also serves the single-shot RAG path whose only time guard is the consumer — so lowering the cap to bound agent turns must not starve a slow single-shot first token. The vault.js abort is computed **live at send time** in `_chatAbortMs()` (reads the `set-agent-wall-clock` control), NOT cached at init, so a change in the Settings modal takes effect on the next Send without reload. `deck.py` passes the cap to `InProcessChatRunner(turn_timeout_s=...)`. The `_CHAT_TOKEN_TIMEOUT_S = 300` constants are now just the default fallback.
- **Local request timeout** (`local_request_timeout_s`, default 0 = leave each path's SDK default): threaded through `core.providers.base.local_request_timeout()` into `OllamaProvider._client()` (ollama streaming + the agent tool path in `local.py`; the client is cached per `(host, timeout)` in `core/providers/ollama.py` so generation reuses one httpx pool — `get_provider()` returns a fresh provider each call) and `get_llm` (LlamaIndex Ollama `request_timeout`, whose own default is 30 s), plus the LM Studio paths. The LM Studio streaming + agent-tool clients go through `core.providers.lms.get_lmstudio_client`, which **caches** the `openai.OpenAI` client per `(base_url, timeout)` (mirroring the Ollama cache — a fresh per-call client leaked an httpx pool), forces **`max_retries=0`** (the SDK otherwise retries a timed-out call up to 2 more times *internally*, each up to the full per-call `timeout`, so one wedged call could block ~3×timeout and blow past the SSE consumer's stall window — the deck-generation random-failure mode; with retries off, one call is bounded by exactly one `timeout` and recovery is owned at the app level: the deckgen per-section retry, the agent loop's deadline), and rounds/floors the timeout to whole seconds (a non-positive value leaves the SDK default, never `0` = time-out-immediately); the agent path's per-call timeout is **quantised to a coarse ceiling ladder** (`core/agent/loop.py::_TIMEOUT_BUCKETS`, item 2.6) so the caches see at most ~8 keys per host instead of one per remaining-second — a wedged call may outlive the deadline by at most the gap to its bucket (one such call max; the loop's deadline gates bound the rest). Both client caches are themselves **bounded LRUs** (`_CLIENT_CACHE_MAX` = 8; evicted clients are not closed in place — a concurrent call may still stream on them — the dropped reference lets GC close them; pinned by `TestClientCacheBound`). Read per call so the knob applies without restart. **Scope: chat/generation only** — embeddings are excluded from *this* knob (bounding indexing batches would cause spurious failures rather than recovery); the **query-path** embed instead has its own independent always-on bound (item 2.1): `Provider.get_embedding(..., request_timeout_s=…)` with `QUERY_EMBED_TIMEOUT_S` = 30 s (`rag/engine.py`) bounds the single retrieval-time embed HTTP call because it runs while holding `_index_mutation_lock`, and the chat path acquires that lock via a 120 s timed `_acquire_retrieval_lock` (clean error, not an indefinite hang — details in `rag/CLAUDE.md` §Lock discipline); indexing embeds (constructed *without* `request_timeout_s`) stay deliberately unbounded. **Vision/OCR is also excluded**: it has its own always-on bound (`vision_timeout_s`, default 120) so indexing-time image/OCR calls are capped independently of this chat knob (see *OCR / Vision Availability Caching → Call bounds*).

## Logging, Deletion & Vault-Write Auditing

- `chatekld.log` is written under `BASE_DIR` (`~/Library/Application Support/ChatEKLD/` on macOS) for both dev and frozen builds. - Every code path that calls `shutil.rmtree(OBSIDIAN_INDEX_DIR)` must first call `log_storage_deletion(reason)` from `core/utils.py` (stack trace to the log + JSON marker at `BASE_DIR/.last_deletion_log`). ## Security Notes

- Local API calls require `X-Requested-With: ChatEKLD`.
- External `Origin` / `Referer` hosts are rejected; with neither header present (normal for PyWebView), the request is accepted only from a loopback `remote_addr`.
- Stored app data lives in an owner-only app data directory.
- Error messages are sanitised (and API-key-redacted) before they reach the UI.
- Rendered-markdown HTML (vault chat, plain chat, refactor previews) passes through the single `ui.js::sanitiseHtml` gate. Its scheme test normalises each attribute value the way the HTML URL parser does — strip tab/LF/CR anywhere, trim C0 controls + space at the ends, entity decoding already done by DOMParser — **before** matching `javascript:`/`vbscript:`/`data:text/html`, so control-character placement (`jav&#9;ascript:`) cannot smuggle a blocked scheme. Pinned by `tests/js/sanitiseHtml.test.js`.
- Streaming routes emit structured JSON error events before `[DONE]` when generation fails mid-stream.

## Build & Distribution

`install_and_build.sh` (macOS-only) is the single build entry point. Cross-cutting invariants:

- **arm64-only.** The PyInstaller invocation passes `--target-arch arm64`, and the script hard-fails on a non-arm64 build host (`HOST_ARCH="$(uname -m)"` guard, right after the Darwin check). Intel is **unsupported as a target**: the native wheels (torch/lancedb/pyarrow/tokenizers) are not reliably cross-buildable to universal2 or x86_64 from an arm64 host, and Rosetta only translates Intel→Apple-Silicon (never the reverse), so an x86_64 artifact would fail on Apple Silicon just as an arm64 one fails on Intel. `--target-arch arm64` also *asserts* every collected binary has an arm64 slice — a foreign-arch dependency now fails the build loudly instead of shipping a broken bundle.
- **Ad-hoc signed, never notarized.** `codesign --force --sign -` only (no Developer ID, no `--options runtime`, no entitlements — the hardened-runtime keys are inert without `--options runtime`, which requires a Developer ID + notarization we deliberately don't do). A locally built bundle launches on the build machine (no `com.apple.quarantine`); a **transferred** copy is Gatekeeper-blocked until the recipient runs `xattr -dr com.apple.quarantine <app>` (documented in the README and inside the `.dmg`).
- **`.dmg` is the delivery wrapper.** The build emits both `ChatEKLD_<date>.app` (run in place) and `ChatEKLD_<date>.dmg` (share). The image stages the `.app` + an `/Applications` drag-target symlink + a generated `READ ME FIRST.txt` (dated app/binary names + the un-quarantine step) + `packaging/seed_models.command` when that file exists (the offline model-seed helper — see below). Both dated artifacts are git-ignored (`ChatEKLD_*.app/`, `ChatEKLD_*.dmg`).
- **Model caches live OUTSIDE the bundle and are seeded per-machine.** tiktoken BPE (`TIKTOKEN_CACHE_DIR`), NLTK punkt/stopwords (`NLTK_DATA_DIR`), and the ~67 MB HF cross-encoder (`~/.cache/huggingface`) are downloaded by `install_and_build.sh` into the **build machine's** dirs — a copy-only `.app` on a fresh Mac has none of them, so its first *offline* index fails (the LlamaIndex `SentenceSplitter` needs tiktoken + NLTK) and its first chat downloads the reranker. The chosen distribution model is a per-machine seed: the DMG's `packaging/seed_models.command` finds the installed app and runs its frozen binary with `--seed-models`. That flag is handled by `launch.py::seed_models()` at module top level **before** the heavy `import webview; from app import app` block (a `sys.argv` membership test, so Finder's legacy `-psn_…` arg can't match it), so seeding needs neither the native webview framework nor the Flask app — it lazily imports the SAME bundled tiktoken/nltk/sentence-transformers the app uses, downloads all three into the exact locations the app reads, then `raise SystemExit(rc)` (rc≠0 if any download failed, so the `.command` can tell the user to re-run online). Requires network once; afterward the app indexes and chats fully offline. Bundling the caches into the `.app` was deliberately rejected (would bloat the read-only bundle + need a copy-on-first-run redirect). Pinned by `test_launch.py` (seed fetches all three, returns non-zero on failure, and the argv branch exits without starting the window).
- **The local model runner is external, not bundled.** The app shells out to a system Ollama / LM Studio (`core/providers/server.py`, resolved against a Homebrew-augmented PATH for the Finder-minimal env). A fresh Mac must install a runner + pull an embed/chat model, or set an online key in `BASE_DIR/.env` (`.env.example` is the template the README points at). First-run guidance surfaces the missing-runner / no-model state through the existing `#runtime-warning` banner (see the `/api/health` A4 note above) — no new UI. (A5, an in-app "set API key" affordance that writes `BASE_DIR/.env`, is deferred — the `.env.example` + README path is the current route.)

## Known Issues / Deferred Work

All low priority; carried from the 2026-05 code review and the 2026-06-10 audit (whose fixes shipped — see `CHANGELOG.md`).

- **Dead Zotero attachments pipeline** — `audit/core/zotero.py` populates `ZoteroItem.attachments` that nothing reads; removal ripples through the vendored SQL/storage plumbing for little gain.
- **Latent online-adapter streaming paths** — `resolve_chat_provider(stream=True)` skips fallback and adapters' `stream()` don't reassemble tool-call deltas; unreachable today (the agent loop always passes `stream=False`).
- **Agent tool-arg validator covers scalars only** — extend `core/agent/tools.py` before adding a tool with `array`/`object` parameters.
- **Dead fields** — `LLMRequest.stop` is *read* by the three online adapters but never *set* by any caller (so it never takes effect; dropping it would require deleting the adapter reads too). (`RagOperationLock._owner_thread` — the former "written but never read" dead field — was dropped in the 2026-07-01 perf pass.)
- **Rerank still runs under `_index_mutation_lock` (W7a, deferred).** The cross-encoder rerank executes inside `engine.query()` while `stream_chat` holds `_index_mutation_lock`, so during active indexing a rerank (~200 ms at pool 50, more at ceiling 200 / on CPU) briefly stalls an `idx.insert` and vice-versa. Moving *only* the rerank outside the lock needs a safe split of the retrieve→rerank→synthesize flow across both vector backends **and** the online/local streaming paths; the 2026-07-01 pass judged that regression risk too high for the small gain (indexing rarely overlaps chat) and deferred it. The retrieval itself genuinely needs the lock; the config-deepcopy half of that workstream (W7b, `load_config_readonly`) shipped.
- **LanceDB MMR / similarity-cutoff scale** — on the lancedb backend MMR runs client-side over an over-fetched pool (approximate, not the native dense-leg semantics), and the dense *score* is `exp(-distance)`, not raw cosine — so the dense-only-no-reranker `similarity_cutoff` (0.25) has a different scale on lancedb than on `SimpleVectorStore`. Ranking parity holds; absolute cutoff semantics differ. Low priority (reranker is on by default).
- **LanceDB orphan from a deleted note in a crash window — partially fixed (2026-07-03).** The drift check now sweeps orphan rows (ids in the table but not the docstore) via `lancedb_list_doc_ids`/`lancedb_delete_ids` whenever it detects `vec_rows > node_count`, and a failed sweep is logged + surfaced instead of silent. Residual gap: offsetting add/delete drift inside one crash window balances the counts, so the branch never fires and those orphans still linger until a full reindex. Rare; self-resolves on a clean full run.
