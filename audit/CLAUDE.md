# Library Audit — Deep Notes

Loaded on demand when working under `audit/`. The two hard invariants (manual-only scan, read-only posture) are restated in the root `CLAUDE.md`; everything here elaborates on them.

- **Manual-only invariant.** No code path outside `POST /api/audit/scan` may call `audit_manager.start_scan()`. `app.py:create_app()` is required to leave the audit `idle` — `test_audit.py::TestFlaskAppBootDoesNotScan` pins this with a `mock.patch.object(audit_manager, 'start_scan')` spy and will fail loudly if a future change adds an auto-scan hook.
- **Read-only posture.** External stores are opened read-only and never mutated. The Zotero SQLite database is opened with `mode=ro` and immediately backed up into an in-memory connection (`audit/core/zotero.py::_open_snapshot`) so the live DB is untouched even if Zotero is running. PDFs are opened with `pikepdf.open(...)` for header/annotation walking only. BibTeX, frontmatter and Finder xattrs are read in place. The only writable file is `BASE_DIR/audit/mapping.json` (manual PDF↔bib overrides + confirmed-no-match list), reached only through `audit/engine/bridge.py::add_match` / `add_no_match` from the `POST /api/audit/mapping` route.
- **Vault path source of truth.** The audit reuses `obsidian_vault_path` from the main config — no separate vault key. Every other subpath (`audit_attachments_subdir`, `audit_biblio_articles_subdir`, `audit_zotero_notes_subdir`, `audit_master_bib_path`) is configurable via the Library Audit settings modal and validated through the same `api/validators.py` helpers the vault routes use (path-traversal rejected, absolute paths rejected, max-length capped). The generic `POST /api/config` strips `audit_*` keys so those validators cannot be bypassed.
- **Settings adapter.** `audit/config.py::load_settings()` builds a kb_harmonizer-shaped `Settings` dataclass from the active papermind config. It raises `AuditConfigError` when `obsidian_vault_path` is empty or does not exist on disk; the route layer surfaces that as a 400.
- **Background lifecycle.** `audit/manager.py::AuditManager` runs one scan thread at a time. State machine: `idle → scanning → done | error | cancelled`. Cancellation is cooperative via `_stop_event` checked between phases (after inventory build, between inventory and duplicates). The pikepdf annotation read on a single PDF is NOT interruptible. A new scan started while the previous is still finalising is stamped with a fresh `_run_id`; the stale completion drops its result on the floor instead of overwriting the live one.
- **Result cache.** Inventory and duplicate sets are held in memory only — no on-disk cache. `GET /api/audit/inventory` and `GET /api/audit/reports/<name>` return 404 until a scan completes. `clear_results()` exists for tests; production code never calls it.
- **Unmapped-PDF annotation precompute.** The `unread_unzoterod` and `read_unzoterod` reports both iterate `inv.bridge.unmapped_pdfs` and need each PDF's annotation count. Computing that lazily inside the `GET /api/audit/reports/<name>` handler opened thousands of PDFs on the request thread (the "Unread PDFs" tab hung on "Loading…" while pikepdf churned). Instead, `inventory.build_inventory` precomputes those counts during the scan — gated by the same `count_annotations` flag, **parallelised** via `ThreadPoolExecutor` (cap `_ANNOTATION_MAX_WORKERS = min(8, cpu_count)`; pikepdf/qpdf releases the GIL so threads scale) — and caches them on `Inventory.unmapped_annotations: dict[Path, AnnotationsResult]`. The route passes that dict into both reports' `find(..., annotations=...)`. `find()` still falls back to a lazy `read_annotations` for any path **missing** from the dict, so a `count_annotations=False` run (empty cache) or a scan cancelled mid-phase (partial cache) degrades to the old per-request read instead of breaking. Progress lines (`Reading annotations for N unmapped PDFs…`, then `…done/total` every 500) flow through `build_inventory(progress_fn=...)` → `manager._emit` into the status feed. Note: the scan's *mapped*-PDF annotation reads (`Record.annotations_count_max`) remain serial — they're the smaller set and not the bottleneck; consolidating them into the same pool is a deferred follow-up. **Sharp edges (by design):** a `count_annotations=False` scan leaves the cache empty, so opening these two tabs falls back to the full serial on-request read — the original "Loading…" stall returns in that mode; and a *cancelled* scan caches only a partial map, so viewing the tabs afterwards lazily reads the un-cached remainder on the request thread (still strictly less work than the pre-optimisation path, but not free).
- **macOS-only Finder tag reads.** `audit/core/finder_tags.py` short-circuits to an empty list on non-Darwin platforms (the `_load_libc()` guard). The audit therefore degrades gracefully on Linux/Windows test runners but the Finder tag dimension is silently empty there.
- **Frontmatter parse warnings are user-visible.** `audit/core/obsidian.py` collects YAML frontmatter-parse failures into a drainable buffer (`drain_parse_warnings()`, deduped, capped at 100); `audit/manager.py::_run` drains it after the inventory phase and emits the first 5 plus a count into the status feed, since a skipped note means its tags are invisible to `note_tag_drift`. Notes are also cached by `(path, mtime_ns)` in `read_note` — `bridge.py` and `inventory.py` both walk `Z_Zotero_Notes` within one scan, so the second walk (and unchanged-note rescans) is free; a cache hit on a malformed note re-records its warning so the problem never silently looks fixed.
- **macOS-only `reveal` endpoint.** `POST /api/audit/reveal` returns 501 on non-Darwin. Paths are bounded to `settings.vault_root` or `settings.zotero_storage` so a malicious request body cannot point `open -R` at arbitrary files. Zotero items open via the `zotero://select/library/items/<key>` URL handler.

## `/api/audit/*` endpoint details

- `GET /api/audit/status` — `{state, error, started_at, finished_at, has_results, has_duplicates, messages[]}`. Polled by the UI only while `state == "scanning"`.
- `POST /api/audit/scan` — body `{count_annotations: bool?, include_duplicates: bool?}`. The ONLY entry point that starts a scan. 409 if one is already in flight, 400 if config is incomplete.
- `POST /api/audit/cancel` — sets the manager's `_stop_event`; the worker checks between phases.
- `GET /api/audit/inventory` — `{summary, records}`. 404 until a scan completes.
- `GET /api/audit/reports/<name>` — `name` ∈ `{note_tag_drift, unread_unzoterod, zotero_unread, read_unzoterod, zotero_no_pdf, duplicates}`.
- `POST /api/audit/mapping` — body `{pdf, citation_key}` for a manual match, or `{pdf, no_match: true}`.
- `POST /api/audit/reveal` — body `{path}` (reveal in Finder), `{open}` (default app), or `{zotero_key}` (zotero://).
