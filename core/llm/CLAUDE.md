## Local adapters

- Ollama resolves bare model names to installed tags (`OllamaProvider.resolve_model`). LM Studio uses raw model IDs from config and `/v1/models`.
- `_LMStudioOpenAI.metadata` (in `core/providers/lms.py`) forces `is_chat_model=True`. LlamaIndex's upstream `is_chat_model()` helper only recognises OpenAI model name patterns, so without the override any non-OpenAI-named LM Studio model (e.g. `google/gemma-4-e4b`) gets routed to the legacy `/v1/completions` endpoint, which LM Studio answers with an empty stream and surfaces as a spurious "No relevant content found" in vault chat.
- The `local` adapter in `core/llm/adapters/local.py` wraps the existing `core.providers.Provider` so the legacy stream-chat path keeps working; on the tool branch it bypasses the flatten-to-prompt path and calls `ollama.chat()` / LM Studio's OpenAI client directly with structured `messages` + `tools`.

## Online adapter parameter contracts

Verified against the live provider docs (platform.openai.com / platform.claude.com / ai.google.dev, 2026-06). When adding a model or knob, keep these in sync:

- **OpenAI o-series reasoning models** (`o1`/`o3`/`o4`/â€¦, detected by `OpenAIProvider._is_reasoning_model` = `o` + digit) reject `temperature`/`top_p` (only the default `1` is accepted) and require `max_completion_tokens` in place of the deprecated `max_tokens` â€” sending the legacy params 400s the whole call. `_common_params` strips the sampling params and renames the token cap for these; `gpt-*` keep `max_tokens` + sampling (also safer for OpenAI-compatible `base_url` servers that may not know `max_completion_tokens`). gpt-5 reasoning variants are deliberately **not** name-detected (a `gpt-5-chat` accepts the sampling params, so prefix-matching would mis-strip it) â€” revisit if a gpt-5 reasoning id enters the curated set.
- **Anthropic caps `temperature` at 1.0** (OpenAI/Gemini allow 2.0). The vault-chat range is 0-2, so `AnthropicProvider._build_payload` clamps to `min(temp, 1.0)` to avoid a 400 against Anthropic only. Anthropic tools use `input_schema` (vs OpenAI's `parameters`) and `tool_choice` is an object (`{"type": "auto"|"any"|"none"}` or `{"type":"tool","name":â€¦}`), built in `_build_payload`; `anthropic-version: 2023-06-01`, `x-api-key` header.
- **Live model listing** (`list_models()` â†’ `core.llm.model_listing.merged_models` + per-adapter `_fetch_live_models`): OpenAI `client.models.list()` (SDK, drop non-chat ids by the `_NON_CHAT_OPENAI_TOKENS` denylist), Anthropic `GET /v1/models` (`data[].id`, `limit` max 1000, all chat), Gemini `GET /v1beta/models` (`models[].name` stripped of the `models/` prefix, filtered to `supportedGenerationMethods` containing `generateContent`, `pageSize` max 1000). All key-gated (no key â‡’ no network â‡’ curated only), short-TTL cached, and never-raising. No pagination is issued because the page-size maxima dwarf each catalog.

## Fallback policy

- Policy comes from `fallback_provider` + `fallback_on` config keys (default `["timeout", "network", "rate_limit", "server_error"]`). Non-transient errors (`auth`, `invalid_request`, `quota`) always surface immediately.
- Hard quota / billing exhaustion is detected by `core.llm.base.looks_like_quota` (OpenAI `insufficient_quota`, Anthropic "credit balance", billing strings) and mapped to the terminal `ErrorCategory.QUOTA` â€” kept out of `fallback_on` and non-retryable â€” so it is not confused with a transient `rate_limit` (e.g. a Gemini per-minute 429 stays retryable).
- The policy is built by `core.llm.policy.parse_policy_from_config(cfg, primary_override=...)` and used by both `rag.summarizer._stream_online` and `rag.engine._OnlineStreamingResponse._stream`. Both streaming sites only fall back **before the first token**; a mid-stream failure after â‰Ą1 token re-raises (the route emits a structured SSE error) rather than re-streaming the whole answer.
- Online generation params: `online_timeout_s` (default 60), `online_max_retries` (default 3, clamped to `[0, 10]` in `factory.py` to mirror the `/api/config` validator so a hand-edited config can't make the retry loop attempt hundreds of round-trips, exponential backoff with jitter via `core/llm/retry.py`), `online_max_tokens` (default 4096 â€” applied when a route does not set max_tokens explicitly).
- Local request timeout: for **streaming/single-shot** local generation the `local` adapter does not honour `LLMRequest.timeout_s`; instead the underlying `OllamaProvider`/`LMStudioProvider` read `local_request_timeout_s` via `core.providers.base.local_request_timeout()` and apply it to their ollama/openai clients + LlamaIndex LLMs (0 = leave SDK default). **Exception â€” the agent tool-call path** (`_generate_with_tools`): it bounds the client by `_effective_local_timeout(request)` = the tighter of `request.timeout_s` and `local_request_timeout_s` (`OllamaProvider._client(timeout=...)` takes an explicit override; the LM Studio branch passes it to `openai.OpenAI(timeout=...)`). This is what lets the agent loop's per-call timeout â€” capped each iteration to the remaining `deadline_monotonic_s` â€” actually interrupt a wedged local backend, since the loop's between-iteration cancel check cannot reach into a blocking HTTP read. The wall-clock cap (`agent_wall_clock_s`, in the route layer) is the coarse backstop above it. See the timeout-chain note in the root `CLAUDE.md` Limits section.
- Live model listing: each online adapter's `list_models()` merges `CURATED_MODELS` with a best-effort, short-TTL-cached, key-gated live fetch from the provider's models endpoint (`core/llm/model_listing.py::merged_models` + per-adapter `_fetch_live_models`, chat-filtered). Curated stays first (and authoritative for `PRICING_TABLE` + default selection); newly-released ids are appended, retired ones simply never appear. Any failure / missing key / offline degrades to curated-only â€” so no-key callers and the hermetic test suite see the pre-fix behaviour, and the dropdown can never empty.

## Usage / cost tracking

- Token / USD-cost usage is recorded by `core.llm.usage.usage_tracker` for every request (online and local). The tracker writes append-only JSONL to `BASE_DIR/llm_usage.jsonl` and exposes a window-scoped summary on `/api/usage`. Parsed disk records are cached by the file's `(size, mtime_ns)` so the UI poll does not re-parse the whole log; usage records carry a per-record `uid` so the in-memory ring and on-disk JSONL never double- or under-count.
- Pricing per million tokens is in `core/llm/usage.py::PRICING_TABLE` and is overridable via the `llm_pricing_overrides` config key. Anthropic figures were verified against platform.claude.com 2026-05 (the Opus tier is $5/$25 since Opus 4.5 â€” NOT the Claude-3-era $15/$75); unknown models cost out at 0.0 rather than a fabricated number.
- `CURATED_MODELS` in each online adapter must list only ACTIVE models â€” retired IDs 404 at the provider. The Claude 3.x IDs were removed 2026-06 for this reason.

## Tool wire format (agent mode)

Each adapter translates the agent's provider-agnostic `LLMRequest.tools` / `tool_history` into its native dialect via helpers in `core/llm/tool_schema.py` (`build_openai_messages` / `build_anthropic_messages` / `build_gemini_contents` plus per-provider `jsonschema_to_*` and `parse_*` parsers). The Anthropic adapter rebuilds `tool_use` / `tool_result` content blocks; Google uses `functionCall` / `functionResponse` parts and synthesises IDs (Gemini ties responses by name, not ID). Adapters expose `supports_tool_use()` and route structured tool calls only when `LLMRequest.tools` is non-empty; the empty default keeps every existing caller on the original code path.
