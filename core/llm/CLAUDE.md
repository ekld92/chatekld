# LLM Provider Layer ‚Äî Deep Notes

Loaded on demand when working under `core/llm/`. The cross-cutting provider invariants (online = chat-only, embeddings local, API keys env-only, per-provider model keys) live in the root `CLAUDE.md`.

## Local adapters

- Ollama resolves bare model names to installed tags (`OllamaProvider.resolve_model`). LM Studio uses raw model IDs from config and `/v1/models`.
- `_LMStudioOpenAI.metadata` (in `core/providers/lms.py`) forces `is_chat_model=True`. LlamaIndex's upstream `is_chat_model()` helper only recognises OpenAI model name patterns, so without the override any non-OpenAI-named LM Studio model (e.g. `google/gemma-4-e4b`) gets routed to the legacy `/v1/completions` endpoint, which LM Studio answers with an empty stream and surfaces as a spurious "No relevant content found" in vault chat.
- The `local` adapter in `core/llm/adapters/local.py` wraps the existing `core.providers.Provider` so the legacy stream-chat path keeps working; on the tool branch it bypasses the flatten-to-prompt path and calls `ollama.chat()` / LM Studio's OpenAI client directly with structured `messages` + `tools`.

## HTTP client caching (W4, 2026-07-01)

Each adapter caches its transport client at **module scope** so consecutive calls reuse one connection pool + TLS session instead of a fresh handshake per round-trip (N per agent turn / deck section). The local providers already did this (`core/providers/ollama.py`, `lms.py::get_lmstudio_client`, keyed by `(host, timeout)`); the online adapters now match:

- **OpenAI** (`_client_cache`, keyed by `(base_url, timeout, organization, key-fingerprint)`): the SDK bakes the key in at construction, so the key's **sha256 fingerprint** is part of the key ‚Äî a rotated key mints a new client, never reuses stale-auth; the key itself is never stored. `_api_key()` still resolves on every call *first*, so the "missing key ‚áí AUTH, no network" contract that gates live model listing is unchanged.
- **Anthropic / Google** (`_client_cache`, keyed by `(base_url, timeout)`): a long-lived shared `httpx.Client`. Auth is a per-request header/param, so the client is **key-agnostic** and rotation-safe (no fingerprint needed). The streaming path wraps the cached client in `contextlib.nullcontext(...)` so only the inner `client.stream(...)` response closes on exit, never the shared client. The short-lived listing client (`_fetch_live_models`, different/shorter timeout) is intentionally left uncached.
- **LM Studio vision** (`services/vision.py::_chat_lm_studio_image`) reuses the cached `get_lmstudio_client` too (a fresh client per image had leaked a pool per scanned-PDF page).

## Online adapter parameter contracts

Verified against the live provider docs (platform.openai.com / platform.claude.com / ai.google.dev, re-verified 2026-07 after the field-reported gpt-5.x/Fable-5 400s). When adding a model or knob, keep these in sync ‚Äî and note the **reactive heal pass** below is the safety net for the next family drift:

- **OpenAI token cap:** `max_completion_tokens` for every model on the official endpoint (`max_tokens` deprecated 2024-09, auto-converted on older models, hard-400 on the o-series **and the gpt-5.x family** ‚Äî "Unsupported parameter: 'max_tokens'"). Only a custom OpenAI-compatible `base_url` (which may predate the new param) keeps legacy `max_tokens` ‚Äî `OpenAIProvider._uses_max_completion_tokens` decides. **Sampling params:** the o-series (`_is_reasoning_model` = `o` + digit) reject `temperature`/`top_p` and are stripped proactively; gpt-5 *reasoning* variants can't be name-detected (`gpt-5-chat` accepts sampling) and are healed reactively instead.
- **Anthropic sampling params:** REMOVED (400) on `claude-fable-5`/`claude-mythos-5`/`claude-opus-4-7*`/`claude-opus-4-8*`/`claude-sonnet-5*` (`_SAMPLING_REMOVED_MODEL_PREFIXES`; "`temperature` is deprecated for this model") ‚Äî `_build_payload` omits them for those families. Older families keep `temperature` clamped to Anthropic's 1.0 cap (OpenAI/Gemini allow 2.0; the vault-chat range is 0-2). Anthropic tools use `input_schema` (vs OpenAI's `parameters`) and `tool_choice` is an object (`{"type": "auto"|"any"|"none"}` or `{"type":"tool","name":‚Ä¶}`); **tools stay in the payload even for `"none"`** ‚Äî once the history holds `tool_use` blocks (the agent loop's forced final turn), omitting them 400s; `anthropic-version: 2023-06-01`, `x-api-key` header.
- **Reactive param heal (both online adapters):** a 400 that names a shape param the payload carries (OpenAI: structured `body.param` or the quoted name in the message; Anthropic: word-boundary match on the backtick-quoted name) is fixed in place ‚Äî `max_tokens` ‚áÑ `max_completion_tokens` renamed, sampling params dropped ‚Äî and the SAME request retried immediately, at most once per param (`_heal_unsupported_param` / `_strip_rejected_sampling_param`). This converts the next provider parameter drift from a hard outage into one logged extra round-trip. Unrelated 400s surface unchanged. Streaming is covered too: a param 400 is the response status, always pre-token, so the heal can never replay yielded text.
- **Gemini:** `generationConfig` still accepts `temperature`/`topP` on 3.x (de-emphasized, not removed ‚Äî checked 2026-07); no proactive strip. A future hard removal is absorbed by extending the heal pattern, not by a hotfix under outage.
- **Live model listing** (`list_models()` ‚Üí `core.llm.model_listing.merged_models` + per-adapter `_fetch_live_models`): OpenAI `client.models.list()` (SDK, drop non-chat ids by the `_NON_CHAT_OPENAI_TOKENS` denylist), Anthropic `GET /v1/models` (`data[].id`, `limit` max 1000, all chat), Gemini `GET /v1beta/models` (`models[].name` stripped of the `models/` prefix, filtered to `supportedGenerationMethods` containing `generateContent`, `pageSize` max 1000). All key-gated (no key ‚áí no network ‚áí curated only), short-TTL cached, and never-raising. No pagination is issued because the page-size maxima dwarf each catalog.

## Fallback policy

- Policy comes from `fallback_provider` + `fallback_on` config keys (default `["timeout", "network", "rate_limit", "server_error"]`). Non-transient errors (`auth`, `invalid_request`, `quota`) always surface immediately.
- Hard quota / billing exhaustion is detected by `core.llm.base.looks_like_quota` (OpenAI `insufficient_quota`, Anthropic "credit balance", billing strings) and mapped to the terminal `ErrorCategory.QUOTA` ‚Äî kept out of `fallback_on` and non-retryable ‚Äî so it is not confused with a transient `rate_limit` (e.g. a Gemini per-minute 429 stays retryable).
- The policy is built by `core.llm.policy.parse_policy_from_config(cfg, primary_override=...)` and used by both `rag.summarizer._stream_online` and `rag.engine._OnlineStreamingResponse._stream`. Both streaming sites only fall back **before the first token**; a mid-stream failure after ‚â•1 token re-raises (the route emits a structured SSE error) rather than re-streaming the whole answer.
- Online generation params: `online_timeout_s` (default 60), `online_max_retries` (default 3, clamped to `[0, 10]` in `factory.py` to mirror the `/api/config` validator so a hand-edited config can't make the retry loop attempt hundreds of round-trips, exponential backoff with jitter via `core/llm/retry.py`), `online_max_tokens` (default 4096 ‚Äî applied when a route does not set max_tokens explicitly).
- Local request timeout: for **streaming/single-shot** local generation the `local` adapter does not honour `LLMRequest.timeout_s`; instead the underlying `OllamaProvider`/`LMStudioProvider` read `local_request_timeout_s` via `core.providers.base.local_request_timeout()` and apply it to their ollama/openai clients + LlamaIndex LLMs (0 = leave SDK default). **Exception ‚Äî the agent tool-call path** (`_generate_with_tools`): it bounds the client by `_effective_local_timeout(request)` = the tighter of `request.timeout_s` and `local_request_timeout_s` (`OllamaProvider._client(timeout=...)` takes an explicit override; the LM Studio branch passes it to `openai.OpenAI(timeout=...)`). This is what lets the agent loop's per-call timeout ‚Äî set each iteration to the *remaining* `deadline_monotonic_s` budget (i.e. the `agent_wall_clock_s` wall clock, **not** `online_timeout_s`, which is online-only) ‚Äî actually interrupt a wedged local backend, since the loop's between-iteration cancel check cannot reach into a blocking HTTP read. So a local agent call's effective bound is `min(remaining wall-clock, local_request_timeout_s)`; the wall-clock cap is the per-call budget itself, not a coarse backstop above a smaller online cap. See the timeout-chain note in the root `CLAUDE.md` Limits section.
- Live model listing: each online adapter's `list_models()` merges `CURATED_MODELS` with a best-effort, short-TTL-cached, key-gated live fetch from the provider's models endpoint (`core/llm/model_listing.py::merged_models` + per-adapter `_fetch_live_models`, chat-filtered). Curated stays first (and authoritative for `PRICING_TABLE` + default selection); newly-released ids are appended, retired ones simply never appear. Any failure / missing key / offline degrades to curated-only ‚Äî so no-key callers and the hermetic test suite see the pre-fix behaviour, and the dropdown can never empty.

## Prompt caching (Track 5.5, 2026-07-04)

- **Anthropic requests always send the system prompt as a block array with one `cache_control: {type: "ephemeral"}` breakpoint** (`anthropic.py::_build_payload`). Anthropic caching is a prefix match over `tools ‚Üí system ‚Üí messages`, so that single breakpoint caches the tool definitions AND system prompt together ‚Äî the stable prefix repeated across agent-loop iterations, deck per-section turns, and same-settings vault chats. Volatile content (retrieved context, question, tool results) rides in `messages`, after the breakpoint. Below the model's minimum cacheable prefix (1024‚Äì4096 tokens depending on family) the marker is silently ignored ‚Äî no write premium ‚Äî so marking short prompts is safe. Max 4 breakpoints per request; we use 1.
- **Usage semantics are normalised at the adapter** (`_usage_from_anthropic_block`): Anthropic's `usage.input_tokens` EXCLUDES cache reads/writes (total prompt = input + cache_read + cache_creation) while OpenAI's `prompt_tokens` INCLUDES its cached subset. `LLMUsage.input_tokens` is therefore always the TOTAL prompt, `cached_input_tokens` the cache-READ subset, and `cache_creation_input_tokens` (new field) the Anthropic cache-WRITE subset. Never record the raw Anthropic numbers ‚Äî the cost formula and /api/usage totals assume the inclusive shape. Pinned by `TestAnthropicPromptCaching`.
- **Pricing**: cache reads bill at `ModelPricing.cached_input` (0.1√ó input for Anthropic entries; OpenAI's published cached rates); cache writes at `ANTHROPIC_CACHE_WRITE_MULTIPLIER` (1.25√ó, the 5-min-TTL premium ‚Äî the 1-h 2√ó tier is unused). `UsageRecord` gained `cache_creation_input_tokens` (defaulted, so legacy JSONL rows still parse).
- **OpenAI**: no request change needed ‚Äî `build_openai_messages` already puts the system prompt first (stable-prefix-first, which is all OpenAI's automatic caching requires) and the adapter already captures `prompt_tokens_details.cached_tokens`.
- **Curated-vs-pricing guard**: every id in the three adapters' `CURATED_MODELS` must have a `PRICING_TABLE` entry ‚Äî a gap silently costs that model out at $0. `usage.unpriced_curated_models()` (adapters imported lazily; they import usage.py) is logged at startup by `app.create_app` and pinned EMPTY by `test_all_curated_models_are_priced`, so adding a curated id without pricing fails the suite.

## Usage / cost tracking

- Token / USD-cost usage is recorded by `core.llm.usage.usage_tracker` for every request (online and local). The tracker writes append-only JSONL to `BASE_DIR/llm_usage.jsonl` and exposes a window-scoped summary on `/api/usage`. Parsed disk records are cached by the file's `(size, mtime_ns)` so the UI poll does not re-parse the whole log; usage records carry a per-record `uid` so the in-memory ring and on-disk JSONL never double- or under-count.
- Pricing per million tokens is in `core/llm/usage.py::PRICING_TABLE` and is overridable via the `llm_pricing_overrides` config key. Anthropic figures were verified against platform.claude.com 2026-05 (the Opus tier is $5/$25 since Opus 4.5 ‚Äî NOT the Claude-3-era $15/$75); unknown models cost out at 0.0 rather than a fabricated number.
- `CURATED_MODELS` in each online adapter must list only ACTIVE models ‚Äî retired IDs 404 at the provider. The Claude 3.x IDs were removed 2026-06 for this reason.

## Tool wire format (agent mode)

Each adapter translates the agent's provider-agnostic `LLMRequest.tools` / `tool_history` into its native dialect via helpers in `core/llm/tool_schema.py` (`build_openai_messages` / `build_anthropic_messages` / `build_gemini_contents` plus per-provider `jsonschema_to_*` and `parse_*` parsers). The Anthropic adapter rebuilds `tool_use` / `tool_result` content blocks; Google uses `functionCall` / `functionResponse` parts and synthesises IDs (Gemini ties responses by name, not ID). Adapters expose `supports_tool_use()` and route structured tool calls only when `LLMRequest.tools` is non-empty; the empty default keeps every existing caller on the original code path.

## Provider wire-format contracts (2026-07-02 fixes)

Field-reported breakage the mocked suite could not see ‚Äî `test_llm.py::TestProviderToolPayloadContract` is the in-process stand-in for provider-side request validation. Keep these invariants when touching the wire layer:

- **Tool names**: OpenAI requires `^[a-zA-Z0-9_-]+$` and Anthropic `^[a-zA-Z0-9_-]{1,128}$` ‚Äî **no dot** ‚Äî while Gemini caps at 64 chars and requires a letter/underscore start. The serializers pass `ToolSchema.name` through verbatim, so an illegal name 400s every tool-enabled request on that provider before any generation (the original dotted `vault.search` names broke agent mode + Deck Generate/Augment on OpenAI *and* Anthropic; Gemini tolerated the dot, which is why only it "worked"). `core/agent/tools.py::ToolRegistry` now enforces the strictest intersection (`VALID_TOOL_NAME_RE = ^[A-Za-z_][A-Za-z0-9_-]{0,63}$`) at construction.
- **Gemini 3.x thought signatures**: Gemini 3 attaches an opaque part-level `thoughtSignature` to each `functionCall` part and REQUIRES it echoed back verbatim on that part in the follow-up request ‚Äî omitting it 400s the second tool turn ("Function call is missing a thought_signature"). `ToolCall.thought_signature` carries it: captured in `google.py::_extract_text_finish_and_calls` (part level, not inside `functionCall`), re-emitted by `build_gemini_contents` (key absent when empty ‚Äî an empty-string signature is not "no signature" to the endpoint). Inert for all other providers and for pre-3.x Gemini models, which never set it.
- **Ollama tool history needs dict arguments**: `build_openai_messages` serialises assistant `tool_calls[].function.arguments` as a JSON *string* (correct for OpenAI / LM Studio), but the ollama client validates outgoing messages against its own pydantic `Message` model whose `ToolCall.Function.arguments` is `Mapping[str, Any]` ‚Äî the string shape made EVERY multi-turn agent conversation on Ollama fail at iteration 2 with a pydantic "Input should be a valid dictionary" error (not a model-capability problem). `local.py::_ollama_messages` parses the strings back to dicts for the ollama branch only and populates ollama's native `tool_name` on `role:"tool"` results (ollama has no `tool_call_id`; its model ignores extra keys).
- **Retry deadlines**: `retry_with_backoff(..., deadline_monotonic_s=)` truncates backoff sleeps to the remaining budget and re-raises instead of starting an attempt with < 0.5 s left. Each online adapter's `generate()` derives the deadline from `request.timeout_s` (the agent loop sets that to its remaining wall-clock each iteration), so a retried transient error can no longer sleep past the turn deadline. `None` (non-agent callers) preserves the old behaviour.
- **Retry-after honoring (2026-07-03)**: 429s carry the provider's required wait ‚Äî the `Retry-After` header and/or the "Please try again in Xs" body phrase. Adapters capture it into `LLMError.retry_after_s` at classification time (`core/llm/retry.py::parse_retry_after_s`); `retry_with_backoff` treats it as a **floor** over the exponential schedule (allowed to exceed `max_delay_s`, capped at `RETRY_AFTER_CEILING_S` = 90 s) because sleeping less is a guaranteed second 429, and raises immediately when the hint exceeds the remaining deadline (every further attempt would be futile). `deckgen/retry.py::chat_with_retry` applies the same floor by regex-parsing the hint out of `ChatResult.error` (that module is pure ‚Äî the structured error never crosses the ChatResult boundary).
- **Forced final agent turn (2026-07-03)**: on the LAST permitted agent iteration (`max_iterations ‚â• 2`), `core/agent/loop.py` sends `tool_choice="none"` plus a "FINAL ITERATION ‚Äî answer now" system-prompt suffix, so a model that spent every iteration on `vault_search` still ends the turn with text instead of the bare "reached the iteration limit" failure that discarded whole deck augments. Per-adapter mapping: OpenAI/LM Studio native `"none"`, Gemini `functionCallingConfig.mode=NONE`, Anthropic `{"type":"none"}` **with tools kept in the payload** (required once the history has `tool_use` blocks), Ollama by omitting `tools` (its API has no tool_choice; `role:"tool"` history is native). A backend that ignores the forcing degrades to the old iteration-limit message.
- **Cost write-back**: `usage_tracker.record()` assigns the computed cost to the passed `LLMUsage.estimated_cost_usd` (the same object every adapter attaches to its `LLMResponse`), which is what makes the agent loop's `UsageBudget` ‚Äî and the per-turn usage footer ‚Äî show real dollars. It also resolves `llm_pricing_overrides` from config when the caller passes none, so the documented override knob actually applies to recorded costs. Unknown models (all local ones) still cost out at 0.0.
