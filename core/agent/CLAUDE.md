# Agent Mode (Vault Chat) — Deep Notes

Loaded on demand when working under `core/agent/`. The opt-in summary (toggle, three tools, additive-only guarantee) lives in the root `CLAUDE.md`.

- Package layout: `protocol.py` (typed `AgentEvent` subclasses), `tools.py` (`ToolRegistry`, `ToolSpec`, `wrap_untrusted`, the tiny JSON-Schema validator), `vault_tools.py` (the three concrete vault tools), `budget.py` (`UsageBudget` per-turn accumulator), `loop.py` (`run_agent_loop` driver + `AgentCapabilityState`). Imports flow agent → llm; the agent layer never reaches into the indexer's internals.
- Three tools are registered: `vault.search(query, top_k≤12)`, `vault.read_note(rel_path)`, and `vault.list_materials(filter?, limit≤200)`. Each is a thin runner over `ObsidianVaultManager` (`retrieve`, `read_note`, `get_indexed_materials`); the manager owns all path-safety, lock discipline, and PDF cache logic. Tool runners return JSON; the registry truncates per-tool output (8 KB default, 32 KB for `read_note`) and `wrap_untrusted` applies the same untrusted-content guard the RAG prompt uses.
- Loop budget: `vault_agent_max_iterations` (default 6, body override clamped to 1–12) per turn. Wall-clock cap is the `agent_wall_clock_s` config key (default 300 s, editable in the LLM Settings window), passed by the route as `deadline_monotonic_s`; `_CHAT_TOKEN_TIMEOUT_S` is now just the fallback default if the key is absent. The loop calls `LLMProvider.generate(stream=False)` for every reasoning/tool iteration — tool-call deltas have to be reassembled before dispatch in all four adapters, so streaming buys nothing for the reasoning phase. The final answer is emitted as a single `TokenEvent` (saves the extra round-trip; revisitable if users miss the typewriter effect).
- Tool exceptions become `ToolResult(is_error=True)` observations rather than killing the turn — the model gets to see "your search failed because X" and recover. `ToolArgError` from schema validation does the same. Provider errors (`LLMError`) DO terminate the turn with an `ErrorEvent` after `core.llm.redact.redact` scrubs any leaked API key.
- Fail-closed fallback: two consecutive iterations where the model produced no parseable tool call (e.g. `finish_reason=TOOL_USE` with empty `tool_calls`, or an empty `STOP`) trigger an `InfoEvent("Agent could not produce a valid tool call …")` followed by a clean re-run of `stream_chat` against the ORIGINAL user message (no agent trace mixed in — avoids feeding partially-corrupted reasoning back into a single-shot prompt). A successful tool call resets the streak counter.
- Capability warning: a module-level `_agent_capability_state` singleton in `api/routes/vault.py` survives across requests within a session. After two consecutive turns each end in fallback, the loop emits a one-shot `InfoEvent` suggesting a tool-capable model (Qwen 2.5, Llama 3.1+, Mistral Nemo, or any online provider). The warning never repeats within a session; a single successful turn resets the consecutive-fallbacks counter to zero.
- System prompt is the user's `vault_chat_system_prompt` (still capped at `VAULT_SYSTEM_PROMPT_LIMIT`) layered on top of a fixed agent preamble that names the tools, steers toward few focused searches with a stop condition + a one-line worked example (2026-06 prompt audit — reduces small-model tool-call looping/malformed-call fallbacks), and reinforces the untrusted-tool-output guard. The user's prompt cannot override the safety preamble.
- Adapter capability probe: `LLMProvider.supports_tool_use()` returns `True` for OpenAI, Anthropic, Google, and Local (both Ollama and LM Studio). Whether the *configured local model* honours the schema is detected at runtime through the malformed-call counter — small Ollama models will silently degrade to RAG, and the capability warning surfaces that. Per-provider wire formats live in `core/llm/tool_schema.py` (see `core/llm/CLAUDE.md`).
- SSE contract additions: `data: {"iteration": N}` / `{"thought": "..."}` / `{"tool_call": {"id","name","arguments"}}` / `{"tool_result": {"tool_call_id","content","is_error","truncated"}}`. The final answer still arrives as `{"token": "..."}`, the existing `{"info": "..."}` / `{"error": "..."}` / `[DONE]` frames are unchanged. After a successful agent turn the route emits one more `info` event with the per-turn `UsageBudget` summary (iteration count + token totals; cost suffix appears only when non-zero). The frontend renders the trace as a collapsed `<details class="agent-trace">` block above the bot bubble.
- Concurrency: each `vault.search` tool invocation acquires `_index_mutation_lock` for the brief retrieval phase (identical to single-shot RAG). The wider agent loop holds no manager-level lock — multiple iterations sequentially re-acquire the mutation lock for each tool call. `RagOperationLock`, the prewarm banner, and the Send-disabled prewarm-in-flight logic are untouched.
- In `api/routes/vault.py::_run_agent`, the `VaultToolContext` fallbacks for `hybrid_enabled` / `reranker_enabled` default `True` to match the documented engine defaults (only reachable if a persisted `vault_*` key is missing or fails coercion — `_resolve_chat_params` otherwise always supplies a value).
