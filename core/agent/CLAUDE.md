# Agent Mode (Vault Chat) — Deep Notes

Loaded on demand when working under `core/agent/`. The opt-in summary (toggle, three tools, additive-only guarantee) lives in the root `CLAUDE.md`.

- Package layout: `protocol.py` (typed `AgentEvent` subclasses), `tools.py` (`ToolRegistry`, `ToolSpec`, `wrap_untrusted`, the tiny JSON-Schema validator), `vault_tools.py` (the three concrete vault tools), `budget.py` (`UsageBudget` per-turn accumulator), `loop.py` (`run_agent_loop` driver + `AgentCapabilityState`). Imports flow agent → llm; the agent layer never reaches into the indexer's internals.
- Three tools are registered: `vault_search(query, top_k≤12)`, `vault_read_note(rel_path)`, and `vault_list_materials(filter?, limit≤200)`. Each is a thin runner over `ObsidianVaultManager` (`retrieve`, `read_note`, `get_indexed_materials`); the manager owns all path-safety, lock discipline, and PDF cache logic. Tool runners return JSON; the registry truncates per-tool output (8 KB `ToolSpec` default, but each vault tool overrides it: 12 KB `vault_search`, 20 KB `vault_list_materials`, ~32 KB `vault_read_note`) and `wrap_untrusted` applies the same untrusted-content guard the RAG prompt uses — with a **per-call nonce'd tag** (`<tool_output-a1b2c3d4 …>`), so vault content containing a literal closing tag cannot escape the wrapper (improvement plan 1.4).
- **Tool-name gate**: `ToolRegistry.__init__` raises `ValueError` for any name outside `VALID_TOOL_NAME_RE` (`^[A-Za-z_][A-Za-z0-9_-]{0,63}$` — the strictest intersection of OpenAI/Anthropic/Gemini rules). Renamed from the dotted `vault.search` forms 2026-07-02: the dot is illegal in OpenAI's and Anthropic's tool-name regex, which 400'd every tool-enabled request on those providers (Gemini tolerated it). See `core/llm/CLAUDE.md` → *Provider wire-format contracts*.
- Loop budget: `vault_agent_max_iterations` (default 6, body override clamped to 1–12) per turn. Wall-clock cap is the `agent_wall_clock_s` config key (default 300 s, editable in the LLM Settings window), passed by the route as `deadline_monotonic_s`; `_CHAT_TOKEN_TIMEOUT_S` is now just the fallback default if the key is absent. **Per-call timeout**: each iteration's `request.timeout_s` is the *remaining* wall-clock budget quantised up to a coarse ceiling bucket (`_quantise_timeout` over `_TIMEOUT_BUCKETS`, item 2.6 — caps the `(host, timeout)` client-cache keyspace at ~8 per host; a wedged call may outlive the deadline by at most the gap to its bucket, once), **not** `online_timeout_s` — that key is online-only (the online adapters bound themselves by their own `self.timeout_s` and ignore `request.timeout_s`), so a slow local model gets the full `agent_wall_clock_s` per call rather than a 60 s cap. Only the local adapter consumes `request.timeout_s` (via `_effective_local_timeout`, further tightened by `local_request_timeout_s` when set); with no deadline (direct/test callers) it is `None` ⇒ local falls back to `local_request_timeout_s` / the SDK default. `online_max_tokens` still caps every reasoning call's output regardless of provider. The loop calls `LLMProvider.generate(stream=False)` for every reasoning/tool iteration — tool-call deltas have to be reassembled before dispatch in all four adapters, so streaming buys nothing for the reasoning phase. The final answer is emitted as a single `TokenEvent` (saves the extra round-trip; revisitable if users miss the typewriter effect).
- **Tool-dispatch deadline (2026-07-04, item 2.4).** The wall clock now also gates tool execution, not just LLM calls: the loop checks `deadline_monotonic_s` **before each tool dispatch** (a turn past its deadline stops mid-batch with the same `ErrorEvent("Agent timed out…")` as the top-of-iteration check — the docstring always promised this check; the code finally has it), and `VaultToolContext.deadline_monotonic_s` carries the same deadline into the tools so long operations can self-limit: the `vault_read_note` runner passes the remaining budget as `read_note(..., time_budget_s=…)`, whose uncached-PDF fresh-extract branch **refuses to start** (explanatory `IOError` → recoverable tool-error observation) when the budget is under `_FRESH_EXTRACT_MIN_BUDGET_S` (20 s). Cache/range-cache hits are always served; `time_budget_s=None` (every non-agent caller) keeps the legacy unbounded contract. Residual: an extract started above the floor can still overrun it — extraction is not interruptible mid-flight. Pinned by `TestToolDispatchDeadline` + `TestReadNoteFreshExtractBudget`.
- Tool exceptions become `ToolResult(is_error=True)` observations rather than killing the turn — the model gets to see "your search failed because X" and recover. `ToolArgError` from schema validation does the same. Provider errors (`LLMError`) DO terminate the turn with an `ErrorEvent` after `core.llm.redact.redact` scrubs any leaked API key.
- Fail-closed fallback: two consecutive iterations where the model produced no parseable tool call (e.g. `finish_reason=TOOL_USE` with empty `tool_calls`, or an empty `STOP`) trigger an `InfoEvent("Agent could not produce a valid tool call …")` followed by a clean re-run of `stream_chat` against the ORIGINAL user message (no agent trace mixed in — avoids feeding partially-corrupted reasoning back into a single-shot prompt). A successful tool call resets the streak counter.
- Capability warning: a module-level `_agent_capability_state` singleton in `api/routes/vault.py` survives across requests within a session. After two consecutive turns each end in fallback, the loop emits a one-shot `InfoEvent` suggesting a tool-capable model (Qwen 2.5, Llama 3.1+, Mistral Nemo, or any online provider). The warning never repeats within a session; a single successful turn resets the consecutive-fallbacks counter to zero.
- System prompt is the user's `vault_chat_system_prompt` (still capped at `VAULT_SYSTEM_PROMPT_LIMIT`) layered on top of a fixed agent preamble that names the tools, steers toward few focused searches with a stop condition + a one-line worked example (2026-06 prompt audit — reduces small-model tool-call looping/malformed-call fallbacks), and reinforces the untrusted-tool-output guard. The user's prompt cannot override the safety preamble.
- Adapter capability probe: `LLMProvider.supports_tool_use()` returns `True` for OpenAI, Anthropic, Google, and Local (both Ollama and LM Studio). Whether the *configured local model* honours the schema is detected at runtime through the malformed-call counter — small Ollama models will silently degrade to RAG, and the capability warning surfaces that. Per-provider wire formats live in `core/llm/tool_schema.py` (see `core/llm/CLAUDE.md`).
- SSE contract additions: `data: {"iteration": N}` / `{"thought": "..."}` / `{"tool_call": {"id","name","arguments"}}` / `{"tool_result": {"tool_call_id","content","is_error","truncated"}}`. The final answer still arrives as `{"token": "..."}`, the existing `{"info": "..."}` / `{"error": "..."}` / `[DONE]` frames are unchanged. After a successful agent turn the route emits one more `info` event with the per-turn `UsageBudget` summary (iteration count + token totals; cost suffix appears only when non-zero). The frontend renders the trace as a collapsed `<details class="agent-trace">` block above the bot bubble.
- Concurrency: each `vault_search` tool invocation acquires `_index_mutation_lock` for the brief retrieval phase (identical to single-shot RAG). The wider agent loop holds no manager-level lock — multiple iterations sequentially re-acquire the mutation lock for each tool call. `RagOperationLock`, the prewarm banner, and the Send-disabled prewarm-in-flight logic are untouched.
- In `api/routes/vault.py::_run_agent`, the `VaultToolContext` fallbacks for `hybrid_enabled` / `reranker_enabled` default `True` to match the documented engine defaults (only reachable if a persisted `vault_*` key is missing or fails coercion — `_resolve_chat_params` otherwise always supplies a value).

## Forced final answer (2026-07-03)

On the LAST permitted iteration (`max_iterations >= 2`), the loop sends `tool_choice="none"` plus a `FINAL ITERATION` system-prompt suffix so a model that spent every iteration on `vault_search` still ends the turn with text (field-reported: the bare "reached the iteration limit" ended deck augments with nothing to apply). Adapter mapping and rationale in `core/llm/CLAUDE.md`; a backend that ignores the forcing (e.g. a weak local model that tool-calls anyway) degrades to the old iteration-limit message. `max_iterations == 1` is exempt — forcing there would turn agent mode into plain no-tool chat on its only call. Pinned by `test_agent.py::TestForcedFinalIteration`.
